🗺️ O Blueprint

Por que o Claudex foi construído em 7 prompts, o que o SPEC.md resolve, e os dois primeiros prompts: esqueleto do plugin e máquina de estados.

O que é:

A maioria das sessões de "construa isso para mim" com Claude falha porque o usuário joga uma spec de 500 linhas em um único prompt e o Claude tenta escrever tudo de uma vez. Cada camada construída em cima de uma camada não verificada é um lugar onde um bug se esconde. O Claudex foi construído incrementalmente porque as garantias do sistema exigem isso — estado precisa funcionar antes de ligar o hook, o hook precisa falhar aberto antes de escrever razões visíveis ao usuário, personas só importam depois que o lifecycle funciona.

Por que aprender:

Entender a filosofia muda como você pensa qualquer build com Claude Code. Cada prompt aqui representa uma fronteira de "verifique antes de continuar". O ponto não é copiar os prompts palavra por palavra — é ver a ordem: esqueleto primeiro, máquina de estados antes do lifecycle, lifecycle antes das personas, personas antes do polish.

Conceitos-chave:

Build incremental · Fronteira de verificação · Camadas dependentes · "Um bug por camada não verificada" · Prompt como unidade de entrega verificável · Fail-open desde o primeiro commit.

O que é:

O SPEC.md é uma referência de arquitetura de página única que o Claude pode consultar a qualquer momento durante o build. Ele define: o que o sistema é, as garantias rígidas (fail-open, sem edições fora do diretório de estado, CAS de fase, escritas atômicas, varredor de loops stale, validação de CWD), o lifecycle completo de fases, o formato do arquivo de estado, os arquivos sidecar, os canais de comunicação entre Claude/Hook/Codex, as personas por rodada, as seis primitivas de segurança e as variáveis de ambiente.

Por que aprender:

Um SPEC bem escrito evita que o Claude precise inventar decisões de design durante o build. Cada prompt dos 7 pede para ler o SPEC.md como fonte da verdade — isso significa que você define uma vez e o Claude consulta várias, em vez de precisar repetir o contexto em cada prompt. O SPEC também funciona como documentação viva: ao terminar o build, ele já descreve o que foi construído.

Conceitos-chave:

SPEC como fonte da verdade única · Definir garantias antes de escrever código · Tabela de fases com quem age e o que o hook faz · Campos do arquivo de estado · Canais de comunicação explícitos · SPEC como documentação viva.

O que é:

O SPEC define seis primitivas de segurança inegociáveis: (1) ERR trap fail-open — primeira linha ativa do hook instala um trap que aprova em qualquer erro não tratado. (2) Escritas atômicas de estado — toda mutação passa por tmp + rename. (3) CAS de transição de fase — só escreve se a fase atual bate com o esperado. (4) Lockfile + liveness de PID — PID armazenado pode ser testado com kill -0. (5) Varredor de loops stale — loops abandonados há mais de 15 minutos são removidos. (6) Validação de CWD — o hook falha aberto se o diretório atual diverge do repo_root salvo no estado.

Por que aprender:

Estas seis garantias definem o contrato de segurança do plugin antes de qualquer linha de código ser escrita. O usuário nunca pode ficar preso num loop quebrado, o estado nunca pode ficar corrompido a meio, dois loops nunca podem competir pela mesma sessão. Entender o "porquê" de cada primitiva permite replicar o padrão em qualquer outro plugin.

Conceitos-chave:

Fail-open como contrato · tmp + rename para atomicidade · Compare-and-swap (CAS) de fase · kill -0 como teste de liveness · CLAUDEX_STALE_MINUTES (padrão: 15) · repo_root como âncora de segurança.

O que é:

O primeiro prompt cria apenas o esqueleto do plugin: manifestos, seis arquivos de comando que só fazem echo "TODO", o hooks.json registrando o Stop hook, e o stop-hook.sh já com ERR trap fail-open mas sem lógica real. O objetivo não é ter algo funcional — é ter algo verificável. O plugin precisa carregar no Claude Code e o hook precisa falhar aberto antes de qualquer lógica ser adicionada.

Por que aprender:

O principal aprendizado aqui é a sequência: manifesto antes de comandos, comandos antes de hooks, hook fail-open antes de hook com lógica. Cada item é testável de forma isolada. O platform-validation.sh criado neste prompt verifica apenas se o hook é executável, retorna JSON válido e falha aberto com entrada inválida — nada mais.

O Prompt (PT-BR):
Prompt 01 — scaffold Cole no Claude Code em uma pasta vazia 
Quero construir um plugin para o Claude Code chamado `claudex`. Leia o `SPEC.md` se ele estiver nesta pasta — ele é a fonte da verdade.

Nesta primeira etapa, crie apenas o esqueleto. Sem lógica real ainda. Quero verificar se o plugin carrega no Claude Code antes de conectar qualquer coisa mais complexa.

Construa:

1. O manifesto do marketplace em `.claude-plugin/marketplace.json` e o manifesto do plugin em `plugins/claudex/.claude-plugin/plugin.json`. Autor: `promptadvisers`. Descrição conforme o spec.
2. Seis arquivos de comandos slash em `plugins/claudex/commands/`: `plan.md`, `review.md`, `status.md`, `doctor.md`, `cancel.md`, `rollback.md`. Por enquanto, cada um apenas dá `echo "TODO"` para confirmar que os comandos aparecem na lista do Claude Code. Use YAML frontmatter correto (`description`, `allowed-tools: Bash`).
3. `plugins/claudex/hooks/hooks.json` registrando um hook `Stop` que chama `${CLAUDE_PLUGIN_ROOT}/hooks/stop-hook.sh` com timeout de 60s e um `statusMessage`.
4. `plugins/claudex/hooks/stop-hook.sh`. CRÍTICO: fail-open em todos os casos desde o início. Instale uma trap ERR que imprime `{"decision":"approve"}` em qualquer erro não tratado e sai com código 0. Corpo padrão por enquanto: ler stdin, registrar "sem lógica ainda", imprimir approve. O usuário NUNCA pode ficar preso por um crash do hook.
5. `plugins/claudex/tests/platform-validation.sh`. Um teste bash simples que verifica se o hook é executável, retorna JSON válido quando alimentado com `{}` no stdin, e falha aberto com entrada inválida. Imprima ✓/✗ coloridos por verificação. Saia com código não-zero em qualquer falha.

Não escreva nenhum código de máquina de estados ainda. Isso é o próximo prompt. Apenas o esqueleto + o hook de fail-open incondicional.

Ao terminar, rode `bash plugins/claudex/tests/platform-validation.sh` e me diga o resultado.
Conceitos-chave:

marketplace.json · plugin.json · YAML frontmatter em commands/ · hooks.json com timeout · ERR trap como primeira linha · platform-validation.sh · Verificar o rascunho antes de continuar.

O que é:

Um Stop hook que retorna BLOCK em caso de erro prende o usuário: a sessão do Claude Code não avança, não há mensagem clara de erro, e o único recurso é matar o processo. Fail-open significa que o hook, em qualquer situação de erro — bug de script, arquivo de estado corrompido, codex não disponível, permissão negada — imprime {"decision":"approve"} e sai com código 0. A funcionalidade do Claudex pode não executar, mas o Claude Code nunca fica preso.

Por que aprender:

Fail-open é um padrão de design para qualquer coisa que intercepta um fluxo crítico do usuário. A ERR trap no início do hook garante isso mesmo para erros não antecipados. Esta escolha é feita antes da primeira linha de lógica real — não é um afterthought, é o contrato de base.

Conceitos-chave:

BLOCK vs APPROVE · ERR trap: trap '... approve; exit 0' ERR · Fail-open como padrão de design · Degradação graciosa · O usuário nunca fica preso.

O que é:

O segundo prompt constrói toda a camada de estado sem tocar no lifecycle do hook. O state-helpers.sh expõe funções para gerar review_ids, ler e escrever campos de estado de forma atômica, fazer CAS de fase, gerenciar lockfiles e varrer loops stale. O start-loop.sh inicia um novo loop parseando flags, recusando loops concorrentes por fase (não por presença de arquivo), e gravando o estado inicial. O lifecycle do hook permanece fail-open stub — a verificação é só da máquina de estados em isolamento.

Por que aprender:

A máquina de estados precisa ser verificada em isolamento antes de ser conectada ao hook. Se o CAS não funciona corretamente, dois caminhos do hook podem avançar a mesma fase ao mesmo tempo. Se as escritas não são atômicas, um crash no meio de uma transição deixa o arquivo de estado corrompido. Testar isso separado não é burocracia — é o que permite confiar no estado quando o lifecycle for adicionado na próxima etapa.

O Prompt (PT-BR):
Prompt 02 — state machine Rode após o Prompt 01 passar 
Precisamos da máquina de estados antes de ligar o lifecycle do hook. O Claudex rastreia cada loop em um arquivo YAML-like em `.claude/claudex/<review_id>.state`. Por loop, nunca compartilhado. Veja o SPEC.md para a lista completa de campos.

Construa:

1. `plugins/claudex/scripts/state-helpers.sh`. Sourceable. Expõe:
   - `claudex_new_review_id` — gera `YYYYMMDD-HHMMSS-XXXXXX` (seis caracteres hex minúsculos no final)
   - `claudex_validate_review_id <id>` — verificação de regex estrita; retorna 0/1
   - `claudex_state_write <arquivo> <conteúdo>` — atômico. Escreve em `${arquivo}.tmp.$$` depois `mv -f`. Nunca arquivos parciais
   - `claudex_state_read_field <arquivo> <campo>` — grep do valor YAML-like
   - `claudex_state_set_field <arquivo> <campo> <valor>` — sed in place via tmp + rename. Também atualiza `last_updated_at` para agora (UTC ISO 8601)
   - `claudex_phase_transition <arquivo> <fase_de> <fase_para>` — compare-and-swap. Só escreve se a fase atual bate com `de`. Retorna 0/1
   - `claudex_lock_write <arquivo>` — escreve `$$`
   - `claudex_lock_is_active <arquivo>` — `kill -0` no PID armazenado
   - `claudex_sweep_stale` — `find -mmin +$CLAUDEX_STALE_MINUTES` (padrão 15) em arquivos `*.state`; remove o state file e seus `.lock`, `-runner.sh`, `-prompt.txt`, e subpasta por loop
   - `claudex_find_active_loop` — imprime o caminho para o state file mais recente (por mtime) ou retorna 1 se não houver

2. `plugins/claudex/scripts/start-loop.sh`. Chamado por `/claudex:plan` e `/claudex:review`. Comportamento:
   - Parseia flags antes do tópico: `--rounds N`, `--from-draft`, `--skip-interview`, `--interviewed`. Rejeita `--rounds 0` e não-inteiros com erro claro. Rejeita `--from-draft` se PLAN.md estiver ausente
   - Recusa iniciar se algum state file existente tiver fase fora de `{done, cancelled, errored}`. Concorrência baseada em fase, NÃO em presença de arquivo. State files completos ficam no disco para auditoria
   - Varre loops stale primeiro (`claudex_sweep_stale`)
   - Gera um `review_id`, escreve o state file inicial atomicamente, escreve um lockfile
   - Imprime instruções iniciais no stdout

3. Estenda `plugins/claudex/tests/platform-validation.sh` com seções que exercitam cada helper.

O lifecycle do hook permanece fail-open stub por enquanto. Estamos apenas verificando a máquina de estados em isolamento.

Rode `bash plugins/claudex/tests/platform-validation.sh` e me diga o que passou.
Conceitos-chave:

state-helpers.sh · review_id no formato YYYYMMDD-HHMMSS-XXXXXX · Atomicidade via tmp+rename · CAS de fase · Concorrência por fase (não por arquivo) · start-loop.sh com flags · Varredor de stale.

O que é:

Escrita atômica significa que o arquivo de estado nunca existe em estado parcialmente escrito. A técnica é sempre escrever em um arquivo temporário (arquivo.tmp.$$) e depois renomear para o destino final com mv -f. Em sistemas de arquivos Unix, rename() é atômico no mesmo filesystem. CAS (Compare-and-Swap) de fase significa que a transição só acontece se a fase atual é exatamente o que esperamos — isso evita que dois caminhos de código avancem a mesma fase ao mesmo tempo.

Por que aprender:

Um arquivo de estado corrompido — lido pelo hook enquanto está sendo escrito — pode fazer o hook falhar de formas imprevisíveis. Sem CAS, se o hook dispara duas vezes rápido (o Claude Code pode fazer isso), ambas as chamadas podem avançar a fase para a mesma fase seguinte, causando uma transição dupla. Entender esses dois padrões é suficiente para proteger qualquer sistema de estado baseado em arquivo.

Conceitos-chave:

tmp + rename = atomicidade no mesmo filesystem · CAS = idempotência na transição de fase · Race condition sem CAS · Nunca usar echo > state direto · Aplicável a qualquer sistema de estado em arquivo.

🧠 O Cérebro do Loop

O Stop Hook lifecycle, o runner script que chama o Codex, extração de findings para economizar tokens e rotação de personas por rodada.

O que é:

O terceiro prompt substitui o stub fail-open por o lifecycle real. O hook agora lê o state file, identifica a fase atual e age de acordo: na fase drafting, verifica se o PLAN.md existe e faz CAS para reviewing, gerando o runner script para o Codex. Na fase reviewing, incrementa a rodada ou conclui o loop. Na fase done, limpa artefatos e aprova. Em qualquer fase desconhecida, aprova.

Por que aprender:

Este é o prompt que transforma o plugin de "esqueleto que carrega" em "sistema que pensa". Toda a lógica de controle do loop vive aqui. O BLOCK diz ao Claude o que fazer a seguir — é o canal de comunicação do hook para o modelo. O APPROVE libera a sessão. Entender essa arquitetura explica por que o Claudex consegue coordenar Claude e Codex sem intervenção humana entre rodadas.

O Prompt (PT-BR):
Prompt 03 — stop hook lifecycle A máquina de estados está verificada. Agora ligue o hook. 
Substitua o stub fail-open em `plugins/claudex/hooks/stop-hook.sh` pelo motor de lifecycle real. Leia o SPEC.md para a tabela de fases.

Construa:

1. Helpers no topo: `log()` adiciona ao `.claude/claudex/log` com timestamp UTC. `approve(reason)` registra e imprime `{"decision":"approve"}`. `block(reason)` registra os primeiros 80 chars, escapa o JSON via python3 com fallback sed, depois imprime `{"decision":"block","reason":...}`.

2. ERR trap incondicional. Primeira linha ativa do script. Captura qualquer coisa não tratada explicitamente e falha aberto.

3. Source do state-helpers.sh (ou fail-open se estiver ausente).

4. Encontra o loop ativo. Se não houver, aprova. Se o formato do review_id for inválido, remove o state file ruim e aprova.

5. Lê campos de estado: mode, phase, round, max_rounds, decision_signal, topic, repo_root, started_at_epoch.

6. Validação de CWD: se repo_root divergir do pwd atual, falha aberto com linha de log.

7. Valida numéricos: round ou max_rounds inválidos caem para 1 / max-padrão.

8. Helper `write_runner_script`: constrói um script bash autocontido em `.claude/claudex/<id>-runner.sh`. O script escreve o prompt num heredoc com `<<'PROMPTEOF'` (aspas simples — crítico: evita expansão shell do conteúdo do prompt) e depois roda `codex exec --dangerously-bypass-approvals-and-sandbox < $PROMPT_FILE`.

9. Case statement para modo plan:
   - `drafting`: se PLAN.md ausente ou vazio → BLOCK pedindo ao Claude para rascunhá-lo. Se presente → CAS para `reviewing`, constrói prompt Codex da rodada 1, escreve o runner, BLOCK com instruções para rodar o runner e ler o arquivo de findings.
   - `reviewing`: se `decision_signal=no-material-findings` → CAS para `done`, limpa, aprova. Caso contrário incrementa rodada; se nova rodada > max_rounds, set `decision_signal=max-reached`, CAS para `done`, limpa, aprova. Caso contrário: escreve próximo runner, BLOCK com novas instruções.
   - `done`: limpa artefatos restantes, aprova.
   - qualquer outra fase: loga fase desconhecida, aprova.

10. Case statement para modo review: `reviewing` escreve runner pedindo ao Codex para revisar o diff atual. `done` limpa, aprova.

11. `plugins/claudex/scripts/mark-done.sh`. Recebe review_id como argumento. Valida o formato. Seta `decision_signal=no-material-findings` e `phase=done`. Atômico.

12. Smoke test em `plugins/claudex/tests/smoke-test.sh`. Sobe um repo git temporário, roda start-loop.sh, escreve PLAN.md falso, dispara o hook diretamente e verifica que faz BLOCK. Verifica que o estado transicionou para `reviewing`. Verifica que um runner script foi criado. Verifica que disparar novamente incrementa a rodada. Verifica que mark-done.sh + hook produz approve. Verifica recusa de loop concorrente.

NÃO chame o Codex de verdade em nenhum teste. Todos os testes fazem stub do runner ou só verificam que o arquivo existe.

Rode os dois scripts de teste. Me diga as contagens.
Conceitos-chave:

log/approve/block helpers · Case statement por fase · write_runner_script · BLOCK como canal de comunicação hook→Claude · mark-done.sh · smoke-test.sh · Stub de runner nos testes.

O que é:

O lifecycle completo tem 4 fases: drafting — Claude escreve o PLAN.md, hook verifica existência e faz CAS para reviewing. reviewing — Claude roda o runner (que chama o Codex), lê os findings, revisa o PLAN.md ou chama mark-done.sh. summarizing — hook faz BLOCK com resumo final, Claude imprime ao usuário, hook aprova no próximo disparo. done — limpa artefatos, aprova.

Por que aprender:

Visualizar o fluxo completo esclarece por que o loop não precisa de intervenção humana entre rodadas. O hook é disparado automaticamente pelo Claude Code ao final de cada turno do Claude. O BLOCK é a forma do hook "falar" com o Claude na próxima rodada. O APPROVE libera a sessão. Entender isso permite depurar o loop quando ele para inesperadamente.

Conceitos-chave:

drafting → reviewing → summarizing → done · Hook disparado no final de cada turno · BLOCK = instrução para o próximo turno · CAS de fase garante linearidade · mark-done.sh como sinal de "sem findings materiais".

O que é:

O runner script é um arquivo bash gerado dinamicamente pelo hook para cada rodada. Ele contém o prompt para o Codex num heredoc com <<'PROMPTEOF' (aspas simples — crítico para evitar expansão shell do conteúdo do prompt) e executa codex exec --dangerously-bypass-approvals-and-sandbox < prompt.txt. O BLOCK diz ao Claude para executar esse script. Após a execução, o Codex terá escrito o arquivo de findings, que o Claude lê para decidir o próximo passo.

Por que aprender:

O runner é o ponto de integração entre os dois modelos. Entender como ele funciona explica por que o Codex não precisa de contexto acumulado — cada rodada é uma chamada fresca com o prompt da rodada mais o PLAN.md atual. A flag --dangerously-bypass-approvals-and-sandbox é necessária porque o Codex está rodando dentro de uma chamada de hook, que não tem contexto interativo para pedir aprovação.

Conceitos-chave:

<<'PROMPTEOF' — aspas simples evitam expansão · codex exec --dangerously-bypass-approvals-and-sandbox · Runner gerado por rodada · Findings escritos pelo Codex · Claude lê findings, não o transcript completo.

O que é:

Em campo, Claude estava queimando contexto lendo transcritos do Codex de 30k+ tokens por rodada. O quarto prompt resolve isso pedindo ao Codex que também escreva um resumo limpo e pequeno dos findings — apenas severidade + descrição de uma linha + recomendação — em .claude/claudex/<id>/findings-round-N.md. O BLOCK instrui o Claude a ler esse resumo em vez do transcript completo, exceto quando precisar de contexto adicional.

Por que aprender:

Este é um padrão de economia de contexto que aparece em qualquer sistema multi-modelo. Modelos grandes processando saídas longas de outros modelos acumulam custo e latência rapidamente. Extrair uma visão resumida das informações relevantes — e passar só essa visão para o modelo seguinte — é uma decisão de arquitetura que escala. O mesmo padrão se aplica a qualquer pipeline onde um modelo revisa a saída de outro.

O Prompt (PT-BR):
Prompt 04 — findings + personas Duas mudanças aditivas 
Duas mudanças, ambas aditivas:

## A. Extração de arquivo de findings

Testes em campo mostraram que o Claude estava queimando contexto lendo transcritos do Codex de 30k+ tokens por rodada. Peça ao Codex que também escreva um resumo limpo de bullets que o Claude lê no lugar.

1. Atualize o prompt Codex da rodada N em stop-hook.sh para incluir este parágrafo no final:

   REQUISITO CRÍTICO DE SAÍDA: Após sua análise completa, escreva um resumo limpo apenas dos findings (severidade + descrição de uma linha + recomendação) no arquivo `<findings_path>`. Use este formato exato: `# Findings rodada N\n\n## Alto\n- <descrição> (<recomendação>)\n## Médio\n...\n## Baixo\n...`. Se não houver findings materiais, o arquivo deve conter exatamente `# Findings rodada N\n\nNenhum finding substantivo.`.

2. O hook constrói o caminho dos findings como `.claude/claudex/<review_id>/findings-round-<round>.md`. Pré-cria a subpasta por loop.

3. O BLOCK diz ao Claude: "Quando o Codex terminar, leia o resumo limpo dos findings em `<caminho>`. Esse arquivo é uma lista curta de bullets. Pule o transcript completo a menos que precise de contexto extra."

4. Adicione `claudex_findings_severity_counts <arquivo>` ao state-helpers.sh. Retorna `high=N medium=N low=N` contando linhas de bullet sob cada header `## Severidade`. Retorna zeros se o arquivo estiver ausente.

5. Na rodada 2+, adicione uma linha "**Rodada anterior:** high=N medium=N low=N" no BLOCK para que o Claude veja a trajetória antes da próxima rodada.

## B. Rotação de personas

Faça cada rodada atacar de um ângulo diferente.

1. Crie `plugins/claudex/scripts/personas.sh`. Sourceable. Duas funções:
   - `claudex_persona_for_round <N>` — imprime uma estância multi-parágrafo descrevendo a persona do revisor para a rodada N.
     - Rodada 1: engenheiro sênior cético. Caça falhas de design e premissas quebradas.
     - Rodada 2: revisor de segurança e integridade de dados. Auth, validação, race conditions, recuperação de falha parcial, segredos, trilhas de auditoria, perda de dados.
     - Rodada 3+: revisor ops/SRE. Segurança de rollback, observabilidade, rollout gradual, version skew, ergonomia de on-call. Na rodada 4+, a estância deve dizer "aprofunde os ângulos anteriores" em vez de repetir a mesma coisa.
   - `claudex_persona_label_for_round <N>` — imprime um label de uma linha para uso nos headers de BLOCK.

2. Em `write_runner_script`, source personas.sh e adicione a estância da persona ao início do prompt Codex.

3. Atualize os headers dos BLOCKs para ler: `### Claudex rodada N de M, <label-da-persona>`.
Conceitos-chave:

findings-round-N.md · Economia de contexto · claudex_findings_severity_counts · Trajetória high/medium/low entre rodadas · personas.sh · Stanza de persona no início do prompt Codex.

O que é:

Cada rodada do Codex ataca o plano de um ângulo diferente: Rodada 1 — engenheiro sênior: falhas de design, premissas quebradas, specs ambíguas. Rodada 2 — segurança e integridade de dados: auth, validação, race conditions, recuperação de falha parcial, segredos, auditoria, perda de dados. Rodada 3+ — ops/SRE: segurança de rollback, observabilidade, rollout gradual, version skew, ergonomia de on-call. As estâncias de persona vivem em scripts/personas.sh e são adicionadas ao início do prompt Codex pelo hook a cada rodada.

Por que aprender:

Sem rotação de personas, o Codex tende a repetir os mesmos tipos de critique em cada rodada — a segunda rodada repete a primeira em vez de ir mais fundo. As personas forçam ângulos diferentes: depois do engenheiro verificar design, o revisor de segurança vai fundo em superfícies de ataque. Depois da segurança, o SRE verifica se o sistema sobrevive a rollback e falha de produção. O resultado é um plano revisado sob três lentes complementares.

Conceitos-chave:

Persona como stanza de instruções · Rodada 1: design · Rodada 2: segurança/integridade · Rodada 3+: ops/SRE · "Aprofundar" vs "repetir" · claudex_persona_label_for_round para headers.

O que é:

Sem a fase de sumarização, o loop terminava silenciosamente — o turno do Claude simplesmente parava sem mensagem de fechamento, deixando o usuário sem saber se algo aconteceu. O quinto prompt adiciona a fase summarizing entre reviewing e done. Nela, o hook faz BLOCK com o resumo final — rodadas executadas, tempo total, tabela de findings por rodada. Claude imprime esse resumo ao usuário, termina o turno, e o próximo disparo do hook faz CAS para done e aprova.

Por que aprender:

A sumarização fecha o loop de feedback. Sem ela, o usuário não sabe quantas rodadas rodaram, quantos findings foram encontrados em cada rodada, ou se o loop terminou por conclusão ou por atingir o máximo de rodadas. Com ela, o loop sempre aterra de forma visível. É um princípio de UX: sistemas automatizados que terminam sem mensagem criam ansiedade e desconfiança.

O Prompt (PT-BR):
Prompt 05 — summarizing phase Sem isso, o loop termina em silêncio 
Adicione uma nova fase entre `reviewing` e `done` chamada `summarizing`. O trabalho desta fase é fazer BLOCK uma vez com um resumo final para que o Claude o imprima ao usuário, depois aprovar no próximo disparo.

## Mudança no lifecycle

Antes:  drafting → reviewing → done
Depois: drafting → reviewing → summarizing → done

`summarizing` é tratada como fase **ativa** pela verificação de concorrência do start-loop.sh (ainda não é terminal).

## Mudanças no hook (stop-hook.sh)

1. Adicione um helper `build_rounds_table <rodada_final>` perto do topo. Itera `1..rodada_final`, lê cada arquivo de findings, chama `claudex_findings_severity_counts`, combina com o label da persona. Retorna linhas como:
   - Rodada 1 (Engenheiro sênior): high=2 medium=4 low=1
   - Rodada 2 (Segurança e integridade): high=2 medium=2 low=0

2. No case `reviewing`, quando `decision_signal=no-material-findings`:
   - CAS `reviewing → summarizing` (NÃO para `done`).
   - Constrói o resumo usando `build_rounds_table` e `format_elapsed`.
   - BLOCK com mensagem markdown intitulada `### Loop do Claudex completo ✓`. Corpo: rodadas executadas, tempo total, tabela de findings, depois "**Imprima este resumo ao usuário para que ele veja que o loop aterrou.** Depois termine seu turno."

3. No case `reviewing`, quando overflow de rodadas:
   - Set `decision_signal=max-reached`.
   - CAS `reviewing → summarizing`.
   - BLOCK com `### Loop do Claudex parado no máximo de rodadas (rodada N de M)`. Inclua a tabela de rodadas, o caminho para o arquivo de findings da última rodada, e três opções explícitas: revisar manualmente, re-rodar com `--rounds N` maior, ou aceitar como incompleto conhecido.

4. Adicione um case `summarizing`:
   - CAS `summarizing → done`.
   - Limpa runner, prompt file, lockfile.
   - Loga tempo decorrido.
   - Aprova.

## Mudança no mark-done.sh

`mark-done.sh` atualmente seta `phase=done` e `decision_signal=no-material-findings`. Mude para **apenas** setar `decision_signal=no-material-findings` e **deixar `phase=reviewing`**. O próximo disparo do hook verá o sinal e dirigirá o caminho de sumarização.

Atualize a linha de echo para dizer "Loop marcado como done. Stop hook entregará o resumo no próximo disparo."
Conceitos-chave:

Fase summarizing · build_rounds_table · Término visível vs silencioso · BLOCK uma vez, APPROVE no próximo · mark-done.sh seta signal, não fase · max-reached com 3 opções explícitas.

O que é:

Quando um processo automatizado termina sem mensagem, o usuário não sabe distinguir entre "terminou com sucesso", "falhou silenciosamente" ou "ainda está rodando". No contexto do Claudex, sem a sumarização o turno do Claude simplesmente parava — nenhum sinal de que 3 rodadas de revisão aconteceram, nenhuma tabela de findings, nenhum link para o PLAN.md final. O sistema parecia quebrado mesmo quando funcionava perfeitamente.

Por que aprender:

Este princípio se aplica a qualquer automação que o usuário não consegue ver executar. Sempre que um processo automatizado conclui — seja um deploy, um batch job, uma revisão de código — ele deve emitir um sinal claro de fechamento com suficiente contexto para o usuário entender o que aconteceu. No caso do Claudex, a fase de sumarização é essa garantia arquitetural.

Conceitos-chave:

Término visível como contrato de UX · Sumarização como fase obrigatória · "Aterrar de forma visível" · Diferença entre sucesso silencioso e falha silenciosa · Resumo como artefato de encerramento.

## Checklist de auditoria

Percorra cada caminho de código e confirme:

1. ERR trap. Primeira linha do stop-hook.sh após as definições de helper instala um trap que retorna approve em qualquer erro e sai com 0. Confirme que nenhum caminho que muta estado roda antes do trap.

2. Escritas atômicas. Toda mutação de estado passa por `claudex_state_write` ou `claudex_state_set_field` (que usam tmp + rename). Nenhum redirecionamento `>` direto para state files em lugar nenhum. Faça grep no codebase para confirmar.

3. CAS de transição de fase. Toda escrita no campo `phase` usa `claudex_phase_transition`, NÃO `claudex_state_set_field "phase"`. Exceção: sites de "força" deliberados como cancel-loop.sh que substituem independente da fase atual. Documente quais sites são quais em comentários de código.

4. Lockfile + liveness de PID. `claudex_lock_is_active` faz `kill -0`. O lockfile é apenas informacional nesta arquitetura — não bloqueia concorrência (a fase faz isso). Não carregue o lock para nada importante.

5. Varredor stale. Roda em cada invocação do start-loop.sh ANTES da verificação de concorrência. Loops completos antigos mantidos no disco para auditoria não são varridos; apenas state files mais velhos que CLAUDEX_STALE_MINUTES.

6. Validação de CWD. Hook lê `repo_root` do estado e falha aberto se `pwd != repo_root`.

📊 O que isso nos deu

Antes e depois do safety pass, bugs capturados pelo processo, o custo real de construir sobre camadas não verificadas.

O que é:

Depois de 6 prompts de construção, o safety pass do Prompt 07 encontrou problemas que passaram despercebidos durante o build incremental: paths de ERR trap que poderiam executar antes da instalação do trap, um redirect direto para um state file que escapou da regra de atomicidade, um site de transição de fase que usava state_set_field em vez de phase_transition. Por que aprender:

O safety pass não é burocracia — é a diferença entre "funciona nos testes" e "funciona quando algo dá errado em produção". Os bugs encontrados pelo safety pass são exatamente os bugs que aparecem em situações de falha parcial: o state file que fica corrompido porque a escrita não era atômica, o loop que avança duas fases porque o CAS não estava no lugar. São bugs difíceis de reproduzir e fáceis de auditar.

Conceitos-chave:

Auditoria sistemática vs revisão ad hoc · Bugs de caminho de erro · Falha parcial como classe de problema · "Funciona nos testes" ≠ "robusto em falha" · Safety pass como última camada.

O que é:

Bugs capturados pelo processo incremental, não pela revisão do código final: a fase de sumarização foi adicionada porque sem ela o loop terminava silenciosamente — problema descoberto ao testar o smoke-test. O heredoc com aspas simples no runner script foi uma correção necessária descoberta ao testar que o Codex recebia o prompt correto — sem aspas simples, variáveis shell no prompt eram expandidas antes de chegarem ao Codex. A validação de review_id contra path traversal foi adicionada ao perceber que um review_id malicioso poderia remover arquivos fora do diretório de estado.

Por que aprender:

Cada um desses bugs teria chegado à produção num build de prompt único porque a superfície de teste seria muito grande para ser coberta manualmente. O build incremental com verificação entre etapas cria uma rede de segurança onde cada etapa só pode introduzir a classe de bugs da sua própria responsabilidade — muito mais fácil de auditar.

Conceitos-chave:

Bugs de silêncio (sem sumarização) · Expansão de variável em heredoc · Path traversal em review_id · Build incremental como rede de segurança · Superfície de teste menor por etapa.

O que é:

O SPEC tem ~500 linhas. Se esse conteúdo fosse passado num único prompt pedindo para construir o plugin completo de uma vez, o Claude produziria código — provavelmente com partes funcionais e partes incorretas misturadas. Identificar o que está errado exigiria ler e entender o output completo de uma vez. As garantias de atomicidade, CAS e fail-open provavelmente não estariam todas corretas — elas são invariantes sutis que emergem do entendimento da arquitetura, não de seguir instruções lineares.

Por que aprender:

O problema do prompt único não é que o Claude não consegue gerar o código — é que você não consegue verificar se o código está correto. Cada bug introduzido no layer 1 se propaga invisível pelos layers 2-7. No prompt único, você só descobre os problemas quando o sistema todo está rodando — na hora mais difícil de depurar.

Conceitos-chave:

Código correto vs código verificável · Invariantes sutis emergem do build incremental · Propagação de bugs entre camadas · "O Claude gera, mas você precisa verificar" · Depuração de sistema completo vs primitivo isolado.

O que é:

Comparação direta entre as abordagens: Um prompt — rápido para iniciar, mas cada camada adiciona dívida técnica não verificada. O custo real aparece nas semanas seguintes em depuração. Sete prompts — mais lento para iniciar, mas cada camada é verificada antes da próxima. O custo de tokens é maior porque há mais chamadas. O custo de tempo de desenvolvimento é menor porque os bugs são encontrados cedo, quando são baratos de corrigir.

Por que aprender:

A intuição de "um prompt é mais rápido" é correta para o início, mas errada para o total. O custo de depurar um bug de CAS que aparece em produção — porque o estado ficou inconsistente numa falha parcial — é muito maior que o custo de ter verificado o CAS no Prompt 02. Este é o cálculo que o build incremental torna explícito.

Conceitos-chave:

Custo antecipado vs custo postergado · Depurar cedo é barato · Dívida técnica não verificada · "Mais rápido de iniciar" ≠ "mais rápido no total" · Tokens de build vs tokens de depuração.

O que é:

Bugs de arquitetura — falhas na máquina de estados, race conditions, ausência de fail-open — são os mais caros de corrigir porque exigem refatoração em múltiplas camadas ao mesmo tempo. Um bug de CAS descoberto depois que o lifecycle, as personas e a sumarização já foram construídas em cima dele exige entender como as camadas superiores dependem do comportamento incorreto, antes de corrigi-lo. Corrigir o mesmo bug no Prompt 02, antes das camadas superiores existirem, é trivial.

Por que aprender:

A regra geral: o custo de corrigir um bug cresce com o número de camadas construídas em cima da camada onde o bug mora. Verificar cada camada antes de adicionar a próxima minimiza esse custo. No contexto do Claude Code, isso significa: teste o que o Claude construiu antes de pedir para ele construir a próxima parte.

Conceitos-chave:

Bugs de arquitetura vs bugs de implementação · Custo cresce com camadas acima · "Verifique antes de continuar" como regra financeira · Refatoração multi-camada é cara · Testar antes de adicionar a próxima camada.

O que é:

Você acabou de ver como o Claudex foi construído — cada camada, cada prompt, cada decisão de arquitetura. Agora a pergunta natural é: como usar essa ferramenta no dia a dia? O curso iClaudex — Planejamento Inteligente com Claude Code + Codex cobre exatamente isso: os 6 comandos, quando usar, exemplos reais, a entrevista de planejamento, e a filosofia do ciclo propor–questionar–melhorar aplicada a projetos de verdade.

Por que aprender:

Saber como uma ferramenta foi construída e saber usá-la bem são habilidades complementares mas distintas. Quem entende a arquitetura interna usa a ferramenta com mais confiança — sabe por que o /cancel preserva o state file, por que o /doctor deve ser o primeiro comando ao ter problemas, por que a entrevista melhora a qualidade do plano final. 🔬 A Metodologia

Os padrões extraídos do build do Claudex — reutilizáveis em qualquer projeto incremental com Claude Code.

O que é:

Antes do Prompt 01, o SPEC.md foi escrito definindo o que o sistema faz, o que ele não faz, as garantias rígidas, o lifecycle, o formato de estado, os canais de comunicação. Por que aprender:

Conceitos-chave:

SPEC como contrato, não documentação · Invariantes antes da implementação · Decisões de design registradas antes do código · "O que não faz" tão importante quanto "o que faz" · SPEC como referência entre prompts.

O que é:

Qualquer sistema que intercepta um fluxo crítico do usuário deve falhar aberto por padrão. No Prompt 01, o hook foi escrito com ERR trap e comportamento fail-open antes de qualquer lógica existir. Isso não foi adicionado no safety pass — foi o ponto de partida. A lógica de lifecycle foi adicionada por cima de um hook que já garantia não travar o usuário.

Por que aprender:

O padrão é: (1) identifique o pior caso de falha — no Claudex, é travar a sessão do Claude Code. (2) Garanta que o sistema nunca cause esse pior caso, mesmo antes da lógica real existir. (3) Adicione a lógica real por cima dessa garantia de base. Isso se aplica a qualquer sistema de interceptação: proxies, middlewares, hooks de CI, validadores de formulário.

Conceitos-chave:

Pior caso de falha → garantia de base · ERR trap como contrato de não-travar · Fail-open como base, não como afterthought · Aplicável a qualquer interceptor · "Nunca prenda o usuário" como invariante.

O que é:

Cada um dos 7 prompts termina com uma instrução de verificação: "rode platform-validation.sh e me diga o que passou", "rode smoke-test.sh e me diga as contagens". Isso não é opcional. Antes de enviar o próximo prompt, você precisa confirmar que a camada atual está funcionando. Se o platform-validation falha no Prompt 02, você não avança para o Prompt 03 — você depura e corrige primeiro.

Por que aprender:

A fronteira de verificação entre prompts é o mecanismo que torna o build incremental confiável. Sem ela, você tem apenas um build em múltiplos prompts — cada camada pode conter bugs que só aparecem quando a camada seguinte depende delas. Com a fronteira, cada camada é uma base verificada sobre a qual a próxima pode ser construída com confiança.

Conceitos-chave:

Verificação obrigatória antes do próximo prompt · Falha na verificação = depura antes de avançar · Cada camada é uma base verificada · O teste é parte do prompt, não separado · "Rode e me diga" como protocolo.

O que é:

O Claudex armazena todo o estado do loop em arquivos: .claude/claudex/<review_id>.state. Não em variáveis de ambiente, não em memória compartilhada, não em flags temporários. Isso significa que o estado sobrevive a crashes, reinicializações e pode ser inspecionado diretamente com cat. O /status apenas lê esses arquivos. O /doctor verifica se o diretório de estado é gravável.

Por que aprender:

Estado implícito — espalhado em variáveis, flags, condições runtime — é impossível de inspecionar e depurar. Estado explícito em arquivo é visível, versionável, pode ser copiado para reprodução de bug e pode ser removido com rm. O padrão se aplica a qualquer sistema de automação de longa duração: o estado que sobrevive entre invocações deve ser explícito e inspecionável.

Conceitos-chave:

Estado em arquivo vs estado em memória · Sobrevive a crashes · Inspecionável com cat · Removível com rm · Por loop, nunca compartilhado · YAML-like: simples de ler e de grep.

O que é:

O platform-validation.sh testa cada função do state-helpers.sh em isolamento: geração de review_id, validação de formato, escrita atômica, CAS, varredor, lockfile. Nenhuma dessas funções é testada apenas como efeito colateral de um teste de sistema maior. Isso significa que quando o smoke-test falha porque a máquina de estados se comporta inesperadamente, o platform-validation já confirmou que os primitivos individuais estão corretos — o bug está na composição, não nas partes.

Por que aprender:

Testar primitivos isolados acelera a depuração ao reduzir o espaço de busca do bug. Se o CAS falha no smoke-test mas passa no platform-validation, o bug não está no CAS — está em como o lifecycle usa o CAS. Se o CAS também falha no platform-validation, o bug está no próprio helper. Essa distinção pode economizar horas de depuração.

Conceitos-chave:

Teste de primitivo vs teste de composição · Falha no primitive test → bug na implementação · Falha no smoke test (com primitive ok) → bug na composição · Espaço de busca reduzido · platform-validation como base da pirâmide.

O que é:

A sequência extraída do Claudex: (1) Escreva o SPEC antes do Prompt 01 — defina o que faz, o que não faz, invariantes. (2) Prompt 1 = esqueleto verificável — carrega sem lógica, falha aberto. (3) Prompt 2 = estado — sem lógica de negócio ainda, só as primitivas de armazenamento. (4) Prompt 3 = lógica de negócio — agora que o estado está verificado. (5) Prompts 4-5 = features aditivas — sem modificar o que já está testado. (6) Prompt 6 = acabamento operacional. (7) Prompt 7 = safety pass — auditoria contra as invariantes do SPEC.

Por que aprender:

Esta sequência não é específica do Claudex — é um padrão para qualquer sistema com estado persistente construído com Claude Code. Esqueleto → Estado → Lógica → Features → Operacional → Safety. Cada etapa tem uma responsabilidade clara e um critério de verificação. Aplicar esse padrão ao próximo projeto reduz o risco de construir sobre bases não verificadas.

Conceitos-chave:

SPEC → Esqueleto → Estado → Lógica → Features → Operacional → Safety · Cada etapa tem critério de verificação · Features aditivas sem modificar o testado · Safety pass como etapa final · Padrão reutilizável para qualquer sistema com estado.

Módulo 6 ~60 min

🚀 Seu Próximo Projeto

Projeto guiado: construa um hook de linting que bloqueia commits com erros — aplicando exatamente a metodologia do Claudex.

O que é:

Para o hook de linting, o SPEC deve responder: O que faz — interceptar commits via Pre-commit hook, rodar o linter configurado, bloquear se houver erros, aprovar se não houver. O que NÃO faz — não corrige os erros automaticamente, não roda em PR reviews, não suporta múltiplos linters em v1. Invariantes — fail-open obrigatório (um crash no hook nunca deve bloquear um commit), o linter deve ser configurável por variável de ambiente, a mensagem de bloqueio deve listar os arquivos com erro.

Por que aprender:

Escrever o SPEC para o hook de linting antes de qualquer código força decisões que parecem óbvias mas não são: o hook deve rodar em todos os commits ou só quando há arquivos alterados de uma certa extensão? O que acontece se o linter não estiver instalado — bloqueia ou aprova? Responder essas perguntas antes do código evita descobri-las pela primeira vez em produção.

Conceitos-chave:

Seções do SPEC: o que faz, o que não faz, invariantes · Hook de linting como exemplo · Decisões de escopo antes do código · "Linter não instalado" como caso de edge a definir · SPEC como filtro de scope creep.

O que é:

Para o hook de linting, a sequência natural é: Prompt 1 — hook com estrutura mínima: lê stdin de evento, instala ERR trap, imprime approve. Nada mais. Verificação: o hook é executável e retorna JSON válido. Prompt 2 — detecta se há arquivos modificados da extensão-alvo. Verificação: testa com lista de arquivos fake. Prompt 3 — roda o linter nos arquivos detectados. BLOCK se erros, APPROVE se limpo. Verificação: testa com arquivo com erro proposital. Prompt 4 — mensagem de erro formatada com lista de arquivos. Verificação: confirma que a mensagem é legível.

Por que aprender:

O critério de corte entre prompts é sempre: "posso verificar que isso funciona corretamente antes de adicionar a próxima parte?". Detector de arquivos sem o linter pode ser testado com arquivos fake. Linter sem mensagem formatada pode ser testado observando o código de saída. Cada corte cria um ponto de verificação independente.

Conceitos-chave:

Critério de corte: "verificável antes da próxima parte?" · Estrutura → Detecção → Lógica → Polish · Teste com dados fake em cada etapa · Verificação antes de avançar · Não combinar lógica de detecção e lógica de execução no mesmo prompt.

O que é:

Para o hook de linting: Esqueleto — hook que carrega, instala ERR trap, aprova sem lógica. Estado — detecta arquivos modificados, registra quais extensões estão em escopo. Para hooks simples, o "estado" pode ser apenas a lista de arquivos passada pelo evento de hook. Lógica — roda o linter, interpreta o código de saída, decide BLOCK vs APPROVE. Segurança — fail-open se linter não instalado, timeout para prevenir travamento, sanitização dos paths de arquivo para prevenir command injection.

Por que aprender:

A sequência não é arbitrária — é a ordem das dependências. A lógica de bloqueio precisa dos arquivos detectados. A segurança precisa da lógica completa para saber o que sanitizar. O esqueleto precisa existir para qualquer outra coisa funcionar. Inverter a ordem — tentar escrever a lógica antes do esqueleto, ou o safety antes da lógica — cria dependências não resolvíveis.

Conceitos-chave:

Dependências determinam a ordem · Esqueleto: carrega + fail-open · Estado: o que o hook precisa para decidir · Lógica: BLOCK vs APPROVE · Segurança: fail-open de dependência, timeout, sanitização · Não existe atalho válido na sequência.

O que é:

Protocolo de verificação para o hook de linting por etapa: Após esqueleto — teste manual: echo '{}' | bash hook.sh retorna {"decision":"approve"}. Após detecção — crie um arquivo .ts fake, simule o evento de hook com esse arquivo na lista, confirme que o hook o detecta. Após lógica — crie um arquivo com erro de lint proposital, confirme que o hook retorna BLOCK com o nome do arquivo. Após segurança — remova temporariamente o linter do PATH, confirme que o hook retorna APPROVE (fail-open).

Por que aprender:

Cada verificação testa exatamente o que o prompt adicionou, não o sistema completo. Isso torna os testes rápidos de escrever e os bugs fáceis de localizar. A verificação do fail-open para linter não instalado é especialmente importante — é o tipo de caso de edge que só aparece em máquinas de CI onde o ambiente é diferente do ambiente de desenvolvimento.

Conceitos-chave:

Verificação focada no que o prompt adicionou · Teste manual rápido antes do teste automatizado · Testar fail-open deliberadamente · Simular ambiente CI (sem linter) · Uma verificação por fronteira de prompt.

O que é:

As três personas aplicadas ao hook de linting: Engenheiro sênior — verifica design: o hook deveria suportar múltiplos linters? A configuração por variável de ambiente é suficiente ou precisa de um arquivo de config? O que acontece com arquivos deletados na lista de modificados? Segurança — verifica superfícies de ataque: os paths de arquivo são sanitizados antes de passá-los ao linter? Um nome de arquivo com ; rm -rf / seria executado? Ops/SRE — verifica operabilidade: como o desenvolvedor sabe por que o commit foi bloqueado? A mensagem de erro é acionável?

Por que aprender:

Mesmo sem o Claudex, você pode aplicar as três personas manualmente ao revisar qualquer sistema que intercepta um fluxo crítico. A pergunta de command injection nos paths de arquivo (persona de segurança) é exatamente o tipo de coisa que um engenheiro esquece ao focar na lógica de negócio. Internalizar as três perspectivas como hábito de revisão melhora a qualidade do código independentemente da ferramenta.

Conceitos-chave:

Personas como lentes de revisão · Command injection em paths de arquivo · Mensagem de erro acionável · Múltiplos linters como escopo v2 · Três perguntas: design sólido? superfície de ataque coberta? operável em falha?

O que é:

Checklist de segurança para qualquer hook ou plugin antes de shipar: (1) Fail-open — um crash do hook nunca trava o usuário? (2) Escritas atômicas — qualquer arquivo de estado usa tmp+rename? (3) Sanitização de input — inputs do usuário e de eventos são sanitizados antes de usá-los em comandos? (4) Timeout — processos externos têm timeout para não travar indefinidamente? (5) Dependência ausente — o hook lida graciosamente com ferramentas opcionais não instaladas? (6) Mensagem de erro — a mensagem de BLOCK é acionável para o usuário? (7) Testes — platform-validation + smoke passam? (8) Doctor — um script de diagnóstico cobre as verificações principais?

Por que aprender:

Este checklist é o safety pass genérico extraído do Prompt 07 do Claudex. Cada item corresponde a uma classe de problema que aparece em produção: fail-open para crashes inesperados, sanitização para inputs maliciosos, timeout para dependências lentas ou travadas. Revisar o checklist antes de shipar custa 10 minutos e evita a classe de bugs mais difíceis de depurar.

Conceitos-chave:

8 itens: fail-open, atomicidade, sanitização, timeout, dependência ausente, mensagem acionável, testes, doctor · Safety pass como checklist reutilizável · 10 minutos antes de shipar vs horas de depuração depois · Generalizável para qualquer hook ou plugin.
