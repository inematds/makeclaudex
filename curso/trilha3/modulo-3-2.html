1

O que é um Git Worktree

Múltiplos checkouts do mesmo repositório simultaneamente

Git worktree é um recurso que permite ter múltiplos diretórios de trabalho vinculados ao mesmo repositório. Cada worktree tem seu próprio HEAD, seu próprio estado de arquivos e pode estar em uma branch diferente — tudo ao mesmo tempo. Para o revisor de PRs, isso significa que enquanto você trabalha na sua branch em ~/projeto, o sistema pode analisar o PR em /tmp/review-PR123 sem qualquer interferência.

💡 Conceito Principal

Um repositório, múltiplos diretórios de trabalho, zero interferência

O Git mantém um único .git/ compartilhado, mas cada worktree tem seu próprio arquivo HEAD, índice de staging e working tree. Modificar arquivos no worktree de revisão não afeta o índice do worktree principal. É como ter dois terminais abertos em diretórios completamente independentes do mesmo repo.

Anatomia do Git Worktree

~/meu-projeto/          ← worktree principal
├── .git/               ← repositório compartilhado
│   ├── HEAD → main
│   └── worktrees/
│       └── review-PR123/
│           └── HEAD → feature/novo-auth
└── src/

/tmp/review-PR123/      ← worktree de revisão (linked)
├── .git               ← arquivo (não diretório!) apontando para .git/worktrees/review-PR123
└── src/               ← estado da branch feature/novo-auth
⚡ Dica Prática

Execute git worktree list no seu repositório para ver todos os worktrees ativos. Se encontrar worktrees de revisões antigas que não foram limpos, use git worktree prune para remover referências órfãs.

2

Criando Worktrees Temporários

git worktree add com path dinâmico e branch remota

A criação de um worktree de revisão segue um padrão fixo: fetch da branch remota, criação do worktree em path temporário, e registro do path no estado YAML. O path usa o número do PR para garantir unicidade mesmo em execuções paralelas.

worktree.sh — função create_worktree()

create_worktree() {
  local pr_number="$1"
  local branch_name="$2"
  local worktree_path="/tmp/review-PR${pr_number}-$(date +%s)"

  # 1. Fetch da branch remota
  git fetch origin "${branch_name}" 2>/dev/null || {
    echo "ERROR: branch '${branch_name}' not found on origin" >&2
    return 1
  }

  # 2. Cria worktree vinculado à branch do PR
  git worktree add "${worktree_path}" "origin/${branch_name}" 2>/dev/null || {
    echo "ERROR: failed to create worktree at ${worktree_path}" >&2
    return 1
  }

  # 3. Registra no estado YAML
  update_state worktree_path "${worktree_path}"
  update_state worktree_branch "${branch_name}"
  update_state status "REVIEWING"

  echo "${worktree_path}"
}
📊 Detalhes do Comando
git worktree add Cria o linked worktree no path especificado
/tmp/review-PR123 Path fora do projeto — garante isolamento de filesystem
origin/feature-branch Branch remota do PR — não cria branch local que precisaria ser limpa
$(date +%s) Timestamp garante unicidade se mesmo PR for revisado duas vezes
⚡ Dica Prática

Use origin/branch-name (remota) em vez de branch-name (local) no worktree add. Isso evita criar uma branch local que precisaria ser deletada manualmente após a revisão. O worktree fica em modo "detached HEAD" rastreando a remota.

3

Limpeza Garantida

O hook que remove o worktree em qualquer saída

Um worktree esquecido ocupa espaço em disco e pode causar conflitos em revisões futuras do mesmo PR. O cleanup precisa acontecer em qualquer forma de saída: sucesso normal, erro do script, SIGTERM do sistema ou SIGINT (Ctrl+C). Isso é implementado com o trap de shell que cobre todos os casos.

cleanup-hook.sh — limpeza garantida via trap

#!/usr/bin/env bash

# Handler de cleanup — chamado em qualquer saída
cleanup_worktree() {
  local worktree_path
  worktree_path=$(read_state worktree_path 2>/dev/null)

  if [[ -n "${worktree_path}" && -d "${worktree_path}" ]]; then
    # Remove o worktree (--force ignora arquivos modificados no worktree)
    git worktree remove --force "${worktree_path}" 2>/dev/null

    # Remove o diretório se git worktree remove falhar
    rm -rf "${worktree_path}" 2>/dev/null

    # Prune referências órfãs no .git/worktrees/
    git worktree prune 2>/dev/null
  fi

  # Marca estado como DONE ou FAILED
  update_state status "${1:-DONE}" 2>/dev/null
}

# Registra o handler para TODOS os tipos de saída
trap 'cleanup_worktree DONE'    EXIT
trap 'cleanup_worktree FAILED'  ERR
trap 'cleanup_worktree FAILED'  SIGTERM
trap 'cleanup_worktree FAILED'  SIGINT

O que o trap cobre

  • ✓EXIT — saída normal ou por exit code
  • ✓ERR — qualquer comando que falha
  • ✓SIGTERM — kill do processo pelo OS
  • ✓SIGINT — Ctrl+C do usuário

O que não é coberto

  • ✗SIGKILL (kill -9) — impossível interceptar
  • ✗Power failure / kernel panic

Para esses casos: o orphan detector no início de cada revisão limpa worktrees antigos.

💡 Por que --force no remove

Se Claude modificou arquivos dentro do worktree durante a análise (esperado — é um sandbox), git worktree remove sem --force vai falhar dizendo que há modificações não commitadas. O --force remove mesmo assim. Isso é intencional: o worktree é descartável, qualquer modificação nele é lixo por design.

4

Estado do Worktree em YAML

Paths, branch name, PR number, status — persistidos atomicamente

O arquivo de estado do worktree persiste todas as informações necessárias para cleanup e retomada. Em caso de falha catastrófica (processo morto, reboot da máquina), o sistema consegue identificar e remover o worktree órfão na próxima execução usando apenas o arquivo de estado.

.claudex/state/review-PR123.yaml

# Estado da revisão em andamento
version: 2                          # CAS — incrementado a cada transição
pr_number: 123
pr_branch: "feature/novo-auth"
pr_title: "Add JWT authentication to API"

# Worktree
worktree_path: "/tmp/review-PR123-1735000000"
worktree_created_at: "2026-04-29T10:30:00Z"

# Máquina de estados
status: "REVIEWING"                 # SETUP | REVIEWING | SUMMARIZING | DONE | FAILED
current_round: 2
max_rounds: 3
current_persona: "Reviewer"

# Findings acumulados
findings_count: 4
findings_file: ".claudex/state/findings-PR123.json"

# Metadados
started_at: "2026-04-29T10:29:55Z"
last_updated: "2026-04-29T10:35:12Z"
💡 Escrita Atômica do Estado

Nunca escreva diretamente no arquivo de estado. Escreva em arquivo temporário e renomeie atomicamente — o rename em sistemas POSIX é atômico e garante que o arquivo de estado nunca é lido em estado parcial: 

update_state() {
  local key="$1" value="$2"
  local state_file=".claudex/state/review-PR${PR_NUMBER}.yaml"
  local tmp_file="${state_file}.tmp.$$"

  # Escreve novo estado no tmp
  yq e ".${key} = \"${value}\"" "${state_file}" > "${tmp_file}"

  # Rename atômico — nunca há arquivo corrompido
  mv "${tmp_file}" "${state_file}"
}
⚡ Dica Prática

O campo worktree_path no estado é a fonte de verdade para cleanup. O script de cleanup deve sempre ler este campo do estado, nunca construir o path dinamicamente — o path inclui timestamp e pode ser diferente de execução para execução.

5

Fail-Safe de Existência

Conflitos de lock e worktrees duplicados

Antes de criar um novo worktree, o sistema verifica se já existe um worktree de revisão anterior que não foi limpo — seja do mesmo PR ou de outro. Esse orphan detector previne conflitos de path e evita consumo crescente de espaço em disco.

Orphan detection e resolução

detect_and_clean_orphans() {
  # Verifica worktrees do git que apontam para /tmp/review-*
  while IFS= read -r line; do
    local worktree_dir
    worktree_dir=$(echo "${line}" | awk '{print $1}')

    if [[ "${worktree_dir}" =~ ^/tmp/review- ]]; then
      local state_file
      state_file=$(find .claudex/state -name "*.yaml" \
        -exec grep -l "worktree_path: ${worktree_dir}" {} \;)

      if [[ -z "${state_file}" ]]; then
        # Worktree sem estado = órfão — remove
        echo "WARN: orphan worktree found at ${worktree_dir}, cleaning..." >&2
        git worktree remove --force "${worktree_dir}" 2>/dev/null
        rm -rf "${worktree_dir}" 2>/dev/null
      fi
    fi
  done < <(git worktree list --porcelain | grep "^worktree ")
}

# Chama antes de criar novo worktree
detect_and_clean_orphans
WORKTREE_PATH=$(create_worktree "${PR_NUMBER}" "${BRANCH_NAME}")
A

Worktree órfão sem estado (processo morto por SIGKILL)

Solução: orphan detector encontra worktrees sem entrada no estado YAML e os remove antes de criar o novo.

B

Revisão paralela do mesmo PR

Solução: timestamp no path garante unicidade. Duas revisões do PR #123 ficam em /tmp/review-PR123-1735000001 e /tmp/review-PR123-1735000099.

C

Lock file do git worktree

Solução: se git detecta que o worktree está "locked" (outra operação em andamento), o sistema aguarda 5s e tenta novamente antes de abortar.

6

Testando o Isolamento

Verificando que modificações no worktree não afetam a branch ativa

O isolamento deve ser testado explicitamente antes de usar o sistema em repositórios de produção. O teste é simples: criar o worktree, modificar um arquivo dentro dele e verificar que a branch principal permanece intacta.

Smoke test de isolamento

#!/usr/bin/env bash
# test-isolation.sh

set -euo pipefail

REPO_DIR="$(pwd)"
WORKTREE_PATH="/tmp/test-isolation-$(date +%s)"

echo "=== Teste de Isolamento de Worktree ==="

# 1. Estado inicial da branch principal
echo "Branch principal: $(git branch --show-current)"
echo "Status inicial:"
git status --short

# 2. Cria worktree de teste
git worktree add "${WORKTREE_PATH}" HEAD
echo -e "\nWorktree criado em: ${WORKTREE_PATH}"

# 3. Modifica arquivo NO WORKTREE
echo "MODIFICATION" >> "${WORKTREE_PATH}/README.md"
echo "test-artifact.tmp" > "${WORKTREE_PATH}/test-artifact.tmp"

echo -e "\nStatus no worktree (deve mostrar modificações):"
git -C "${WORKTREE_PATH}" status --short

echo -e "\nStatus na branch principal (deve estar LIMPO):"
git -C "${REPO_DIR}" status --short

# Verifica que está limpo
if git -C "${REPO_DIR}" status --short | grep -q .; then
  echo "FAIL: branch principal foi contaminada!" >&2
  exit 1
fi

echo "PASS: branch principal permanece limpa"

# 4. Cleanup
git worktree remove --force "${WORKTREE_PATH}"
echo "Worktree removido. Teste completo."
💡 O que o teste confirma
  • •Arquivos modificados no worktree não aparecem no status da branch principal
  • •Arquivos criados no worktree não aparecem no working tree principal
  • •Após remoção do worktree, nenhum rastro permanece no repositório principal
  • •git worktree list não mostra mais o worktree removido
