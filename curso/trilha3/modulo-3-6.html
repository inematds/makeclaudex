O Safety Pass do Plugin de Revisão

As 6 primitivas aplicadas ao novo contexto

As 6 primitivas de segurança que você aprendeu na Trilha 2 são universais — mas seus pontos de aplicação mudam conforme o sistema. O safety pass consiste em percorrer cada primitiva e verificar onde ela se aplica no revisor de PRs, garantindo que nenhuma foi esquecida.

1

Fail-Open

Novo ponto: O hook de Stop deve retornar exit 0 (APPROVE) em caso de erro ao ler o estado, não exit 2 (BLOCK). Um revisor que bloqueia Claude indefinidamente é pior que não existir.

2

Estado Atômico

Novo ponto: Dois arquivos de estado agora (review YAML + findings JSON). Ambos devem usar escrita atômica via arquivo temp + rename. O worktree_path no YAML é especialmente crítico.

3

Compare-and-Swap (CAS)

Novo ponto: O campo version no YAML deve ser incrementado em cada transição de estado. Antes de escrever, verificar se version == expected_version para evitar escritas concorrentes.

4

Cleanup Garantido

Novo ponto: Dois recursos para limpar: worktree em /tmp E lockfile de revisão. O trap deve remover ambos. A ordem importa: remover worktree antes de limpar estado (estado é prova de o que limpar).

5

Idempotência

Novo ponto: Rodar /review-pr 123 duas vezes não deve criar dois comentários no PR. O marcador garante upsert de comentário. A criação de worktree deve verificar existência primeiro.

6

Timeout

Novo ponto: Cada chamada ao claude --print deve ter timeout explícito (ex: 120s). Uma chamada que não termina bloqueia o loop indefinidamente. Use timeout(1) do sistema ou --timeout flag do CLI.

💡 A Checklist do Safety Pass

Faça o safety pass como o último passo antes de declarar o sistema pronto para produção. Para cada primitiva, responda: "Onde no código esta primitiva é aplicada? Existe um teste que verifica que ela funciona?" Se a resposta para qualquer primitiva for "não sei" ou "não", o sistema não está pronto.

2

Testando com PRs Reais

E2E sintético com um PR de teste com issues conhecidas

Antes de usar o sistema em PRs de produção, valide-o com um PR sintético que contém issues intencionais: um bug, uma vulnerabilidade de segurança e um problema de performance — um para cada persona. Se o sistema encontrar todos os três, as personas estão funcionando corretamente.

PR sintético de teste — código com issues plantadas

// test-pr/vulnerable-api.ts
// Issues plantadas para validação do revisor

import { db } from './db';

// ISSUE 1 (Author/MEDIUM): Função promete retornar User mas pode retornar null
// sem indicar isso no tipo de retorno
export async function getUser(userId: string) {
  const user = await db.query(
    // ISSUE 2 (Security/CRITICAL): SQL Injection — userId não sanitizado
    `SELECT * FROM users WHERE id = '${userId}'`
  );
  return user;
}

export async function getAllUsers() {
  // ISSUE 3 (Reviewer/HIGH): Sem paginação — pode retornar milhões de registros
  // Também: sem autenticação/autorização verificada
  const users = await db.query('SELECT * FROM users');
  return users;
}

// Descrição do PR (test): "Add user lookup endpoints"
// O PR não menciona as issues — Author vai encontrar o gap de intenção

3

Rate Limiting e Custo

Controlando chamadas ao Claude, estimativa por PR

Um plugin sem controle de custo pode gerar faturas inesperadas em repositórios com muitos PRs. O modelo de custo é previsível: custo por PR = (tokens de contexto + tokens de diff) × número de rodadas × preço por token. Configurações de controle transformam isso de surpresa em decisão.

📊 Estimativa de Custo por PR
PR pequeno (100 linhas de diff) ~500 tokens/rodada × 3 rodadas
PR médio (500 linhas de diff) ~2000 tokens/rodada × 3 rodadas
PR grande (2000 linhas de diff) ~8000 tokens/rodada × 3 rodadas

* Estimativas incluem contexto do PR + instrução de persona + findings anteriores. Preço depende do modelo escolhido.

Controles de custo em review-config.yaml

# .claude/review-config.yaml

# Controle de custo
max_diff_lines: 2000          # PRs maiores são revisados por amostragem
max_tokens_per_round: 4096    # Limita output do modelo por rodada
max_rounds: 3                 # Terminação garantida

# Rate limiting
max_reviews_per_hour: 10      # Evita explosão de custo em repos de alto volume
review_throttle_seconds: 30   # Pausa entre revisões consecutivas

# Amostragem para PRs grandes
sample_large_prs: true
sample_strategy: "changed_files_priority"  # Prioriza arquivos mais modificados
⚡ Dica Prática

Adicione um campo estimated_cost_usd ao frontmatter do relatório. Depois de um mês, some os custos por repositório para ter dados reais que justificam (ou ajustam) o investimento em revisão automatizada.

4

Configuração por Repositório

.claude/review-config.yaml com thresholds e personas

Cada repositório tem perfil de risco diferente. Um serviço financeiro precisa de Security como primeira persona e MAX_ROUNDS=5. Um repositório de documentação precisa de Reviewer básico com MAX_ROUNDS=1. A configuração por repositório permite adaptar o sistema sem modificar código.

Repo de serviço financeiro

max_rounds: 5
personas:
  - round: 1
    name: "Security"
  - round: 2
    name: "Author"
  - round: 3
    name: "Reviewer"
  - round: 4
    name: "Security"   # Segunda passagem
  - round: 5
    name: "SRE"

blocking_severity: "MEDIUM"  # Mais restritivo

Repo de frontend/design

max_rounds: 2
personas:
  - round: 1
    name: "Author"
  - round: 2
    name: "Reviewer"

blocking_severity: "HIGH"    # Padrão

# Ignora arquivos não-código
exclude_patterns:
  - "*.svg"
  - "*.png"
  - "*.json"
  - "*.lock"

Carregando configuração no runner

load_config() {
  local config_file=".claude/review-config.yaml"

  # Se não existe config, usa defaults
  if [[ ! -f "${config_file}" ]]; then
    MAX_ROUNDS=3
    BLOCKING_SEVERITY="HIGH"
    return
  fi

  MAX_ROUNDS=$(yq e '.max_rounds // 3' "${config_file}")
  BLOCKING_SEVERITY=$(yq e '.blocking_severity // "HIGH"' "${config_file}")
  MAX_DIFF_LINES=$(yq e '.max_diff_lines // 2000' "${config_file}")

  # Lê personas em ordem de round
  PERSONAS=()
  while IFS= read -r persona; do
    PERSONAS+=("${persona}")
  done < <(yq e '.personas[].name' "${config_file}")
}
💡 Defaults sensatos

O sistema deve ter defaults sensatos que funcionam sem configuração. Um novo repositório sem .claude/review-config.yaml deve ter a revisão padrão (3 rodadas, Author+Reviewer+Security, HIGH como blocking) funcionando imediatamente. Configuração é para ajuste fino, não para funcionamento básico.

O que Separa Hobby de Produção

Observabilidade, rollback e alertas

Um sistema funcional 95% do tempo que falha silenciosamente no resto é mais perigoso que não existir. Em produção, você precisa saber quando o sistema falhou, ser capaz de desfazer suas ações e ter alertas proativos antes que o problema afete desenvolvedores.

OB

Observabilidade — Logs Estruturados

Cada evento do ciclo de vida deve ser logado em JSON com timestamp, PR number, status e duração. Isso permite queries como "todas revisões que demoraram mais de 5 minutos" ou "PRs onde o cleanup falhou".

{"ts":"2026-04-29T10:30:00Z","event":"review.started","pr":123,"worktree":"/tmp/review-PR123-1735000000"}
{"ts":"2026-04-29T10:35:12Z","event":"review.round.completed","pr":123,"round":1,"persona":"Author","findings":2}
{"ts":"2026-04-29T10:45:33Z","event":"review.completed","pr":123,"duration_s":930,"total_findings":6}
RB

Rollback — Desfazendo Comentário Postado

Um relatório com findings falsos positivos pode causar ruído ou bloquear merge indevidamente. O rollback remove o comentário do GitHub e restaura o PR ao estado "sem revisão".

rollback_review() {
  local pr_number="$1"
  # Encontra e deleta o comentário de revisão
  local comment_id
  comment_id=$(gh api "/repos/{owner}/{repo}/issues/${pr_number}/comments" \
    --jq '.[] | select(.body | startswith("")) | .id')
  [[ -n "${comment_id}" ]] && gh api --method DELETE \
    "/repos/{owner}/{repo}/issues/comments/${comment_id}"
}
AL

Alertas — Detectando Falhas Silenciosas

Verifique periodicamente: PRs abertos há mais de X horas sem revisão (revisor falhou silenciosamente), worktrees órfãos acumulando (cleanup não funcionou), custo diário acima do threshold.

⚠️ Teste de Falha Silenciosa

Antes de declarar o sistema em produção, faça um teste de falha silenciosa: mate o processo de revisão com SIGKILL durante a análise e verifique que: 1) o worktree foi limpo na próxima execução, 2) o estado ficou em FAILED (não em REVIEWING), 3) um alerta foi gerado. Se qualquer um falhar, você tem uma falha silenciosa.

Próximos Passos

Integração com CI, revisão multi-model e auto-apply de fixes

O sistema atual é manual — iniciado pelo desenvolvedor via /review-pr. As três extensões naturais o tornam proativo e totalmente integrado ao workflow de engenharia sem exigir ação humana.

🤖

CI Integration

GitHub Action que roda /review-pr automaticamente na abertura de PR ou em push para branches de feature.

on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: /review-pr ${{ github.event.pull_request.number }}
🔄

Revisão Multi-model

Usar modelos diferentes por persona: claude-opus para Security (mais profundo), claude-haiku para Author (mais rápido, mais barato para contexto).

personas:
  - round: 1
    name: "Author"
    model: "claude-haiku-4-5"
  - round: 2
    name: "Reviewer"
    model: "claude-sonnet-4-6"
  - round: 3
    name: "Security"
    model: "claude-opus-4-5"
🔧

Auto-apply de Fixes

Para findings LOW e INFO, aplicar automaticamente as correções sugeridas via patch, criando um commit de "automated-fixes" no PR.

auto_apply:
  enabled: true
  max_severity: "LOW"
  require_review: true  # Claude cria commit,
                        # humano aprova
