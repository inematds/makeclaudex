O Formato do Relatório

Markdown estruturado legível por humanos e por CI/CD

O relatório tem dois públicos: o desenvolvedor que precisa entender o que consertar e o sistema de CI/CD que precisa decidir se permite o merge. Isso significa que o formato precisa ser legível por humanos e parseável por scripts ao mesmo tempo — e Markdown com headers consistentes atende ambos.

Exemplo de relatório gerado (.claudex/reviews/PR-123.md)

---
pr_number: 123
pr_title: "Add JWT authentication to API"
review_date: "2026-04-29T10:45:00Z"
total_findings: 6
critical: 1
high: 2
medium: 2
low: 1
info: 0
reviewer_verdict: "CHANGES_REQUIRED"
---

# Revisão de PR #123: Add JWT authentication to API

## Sumário Executivo

O PR implementa autenticação JWT mas apresenta **3 vulnerabilidades de segurança
encadeadas** que juntas permitem bypass completo do sistema de autenticação.
O problema central é a validação insuficiente de tokens em múltiplos pontos.

**Veredicto:** ❌ CHANGES REQUIRED — 1 finding CRITICAL, 2 findings HIGH

---

## Findings Críticos

### 🔴 CRITICAL — src/auth/middleware.ts:45
**Tipo:** missing-authz | **Persona:** Security
```
JWT token não é validado para operações de admin. A verificação verifica
apenas se o token está presente, não se tem a role 'admin'.
```
**Impacto:** Qualquer usuário com token válido pode executar operações admin.
**Sugestão:** Adicionar check de role após validação de token.

---

## Findings de Alta Severidade

### 🟠 HIGH — src/auth/jwt.ts:78
...

## Métricas

| Métrica | Valor |
|---------|-------|
| Linhas no diff | 342 |
| Arquivos modificados | 8 |
| Rodadas de análise | 3 |
| Personas usadas | Author, Reviewer, Security |
💡 O Frontmatter YAML

O bloco --- no topo é YAML frontmatter. Sistemas de CI/CD podem parsear apenas esse bloco com python-frontmatter ou yq para verificar critical > 0 sem precisar ler o relatório completo. Humanos ignoram o frontmatter e leem o Markdown normalmente.

2

Severidade dos Achados

CRITICAL, HIGH, MEDIUM, LOW, INFO — critérios e exemplos

Severidade sem critérios claros é subjetiva — e subjetividade cria inconsistência entre revisões. Com critérios explícitos, qualquer persona em qualquer rodada classifica o mesmo problema da mesma forma, tornando os relatórios comparáveis ao longo do tempo.

CRITICAL

Bloqueia merge — deve ser resolvido antes de qualquer deploy

Exemplos: bypass de autenticação, SQL injection, exposição de credenciais, possibilidade de execução remota de código, perda de dados irrecuperável.

HIGH

Deve ser resolvido antes de deploy para produção

Exemplos: validação faltando em input público, privilege escalation parcial, dados sensíveis em logs, race condition que corrompe estado.

MEDIUM

Recomendado resolver — pode ser em PR de follow-up

Exemplos: tratamento de erro incompleto, cobertura de teste insuficiente em paths críticos, performance degradada sob carga, falta de rate limiting.

LOW

Sugestão de melhoria — resolve quando conveniente

Exemplos: código duplicado, naming confuso, documentação desatualizada, oportunidade de simplificação.

INFO

Observação informativa — não requer ação

Exemplos: abordagem alternativa que poderia ter sido usada, contexto técnico relevante, inconsistência com padrão do time que pode ser intencional.

⚡ Dica Prática

Configure blocking_severity: "HIGH" no review-config.yaml para que o CI/CD bloqueie merge apenas em CRITICAL e HIGH. Isso evita paralisia por findings de baixa severidade enquanto garante que problemas sérios não chegam a produção.

Agrupamento por Arquivo e Tipo

Organizando achados de múltiplas rodadas para triagem eficiente

Findings listados em ordem cronológica de análise (rodada 1, rodada 2...) forçam o desenvolvedor a pular de um arquivo para outro durante a triagem. Agrupamento por arquivo permite abrir cada arquivo uma vez e resolver todos os problemas juntos.

report.sh — função de agrupamento e ordenação

generate_grouped_report() {
  local findings_file="$1"

  # Extrai todos os findings de todas as rodadas
  # Ordena por: severidade (CRITICAL primeiro) depois por arquivo
  local all_findings
  all_findings=$(jq -r '
    [.rounds[].findings[]] |
    sort_by(
      (if .severity == "CRITICAL" then 0
       elif .severity == "HIGH" then 1
       elif .severity == "MEDIUM" then 2
       elif .severity == "LOW" then 3
       else 4 end),
      .file,
      .line
    )
  ' "${findings_file}")

  # Gera seção por arquivo
  local current_file=""
  echo "${all_findings}" | jq -c '.[]' | while read -r finding; do
    local file
    file=$(echo "${finding}" | jq -r '.file')

    if [[ "${file}" != "${current_file}" ]]; then
      current_file="${file}"
      echo ""
      echo "### 📄 ${file}"
      echo ""
    fi

    # Formata finding individual
    format_finding "${finding}"
  done
}

Ordem de severity

1.CRITICAL — precisa de ação imediata
2.HIGH — bloqueia deploy
3.MEDIUM — próximo sprint
4.LOW — quando conveniente
5.INFO — para contexto

Agrupamento secundário

→Por arquivo (path completo)
→Por número de linha dentro do arquivo
→Opcionalmente: por tipo de issue
💡 Por que agrupamento importa

Um PR com 6 findings em 4 arquivos gera 6 contextos de "abrir arquivo, localizar linha, entender, fechar". Com agrupamento, cada arquivo é aberto uma única vez. Para PRs com 20+ findings, a diferença em produtividade de triagem é de 3-4x.

O Relatório como Artefato Persistente

Salvando em .claudex/reviews/PR-NNN.md para histórico auditável

Relatórios persistidos criam um histórico auditável de revisões. Quando um PR passa por múltiplas iterações (desenvolvedor corrige, cria nova versão, revisão nova), você pode comparar os relatórios para verificar se os findings foram realmente resolvidos.

Convenção de nomenclatura e salvamento

# Estrutura de diretório de relatórios
.claudex/
└── reviews/
    ├── PR-123-v1-20260429.md    # Primeira revisão do PR #123
    ├── PR-123-v2-20260501.md    # Segunda revisão (após correções)
    ├── PR-124-v1-20260430.md
    └── index.yaml               # Índice de todos os relatórios

# Geração com versionamento
save_review_report() {
  local pr_number="$1"
  local report_content="$2"

  # Versão = quantas revisões do mesmo PR já existem + 1
  local version
  version=$(ls .claudex/reviews/PR-${pr_number}-v*.md 2>/dev/null | wc -l)
  version=$(( version + 1 ))

  local report_file=".claudex/reviews/PR-${pr_number}-v${version}-$(date +%Y%m%d).md"

  # Salva o relatório
  echo "${report_content}" > "${report_file}"

  # Atualiza índice
  yq e ".reviews += [{\"pr\": ${pr_number}, \"version\": ${version}, \"file\": \"${report_file}\", \"date\": \"$(date -I)\"}]" \
    .claudex/reviews/index.yaml -i

  echo "${report_file}"
}

Usos do histórico

  • • Comparar v1 vs v2 após correções
  • • Auditar que CRITICAL foi resolvido
  • • Medir qualidade de revisão ao longo do tempo
  • • Detectar regressões (finding reaparece)

O relatório no git

  • • Commitar .claudex/reviews/ no repositório
  • • Histórico de revisões vira parte do repo
  • • git blame no relatório mostra quando finding foi introduzido
  • • PR de "resolução" pode referenciar o finding original

Integração com GitHub

Comentários automáticos via gh pr comment

Um relatório que fica só em arquivo local tem impacto zero. A integração com GitHub coloca os findings onde o desenvolvedor já está — na thread do PR — sem que ele precise procurar o arquivo de relatório.

Postando o comentário no PR

post_review_comment() {
  local pr_number="$1"
  local report_file="$2"

  # Verifica se já existe comentário de revisão anterior no PR
  local existing_comment_id
  existing_comment_id=$(gh pr view "${pr_number}" \
    --json comments \
    --jq '.comments[] | select(.body | startswith("")) | .id' \
    2>/dev/null | head -1)

  if [[ -n "${existing_comment_id}" ]]; then
    # Atualiza comentário existente em vez de criar novo
    gh api \
      --method PATCH \
      "/repos/{owner}/{repo}/issues/comments/${existing_comment_id}" \
      --field body="$(cat "${report_file}")"
    echo "Updated existing review comment ${existing_comment_id}"
  else
    # Cria novo comentário
    gh pr comment "${pr_number}" \
      --body "$(cat "${report_file}")"
    echo "Created new review comment"
  fi
}
💡 O Marcador HTML Oculto

O comentário HTML <!-- claude-review --> é invisível no GitHub mas permite que o script identifique comentários criados pelo revisor automático. Isso evita criar múltiplos comentários em revisões repetidas — o segundo run atualiza o comentário existente.

⚡ Dica Prática

Use gh pr review --request-changes em vez de apenas comentar quando há findings CRITICAL ou HIGH. Isso coloca o PR em estado "Changes Requested" no GitHub, prevenindo merge acidental por outro membro do time antes dos problemas serem resolvidos.

Métricas de Revisão

Linhas analisadas, achados por rodada e cobertura de diff

Métricas de revisão servem dois propósitos: transparência (o desenvolvedor vê o escopo da análise) e calibração (você pode ajustar MAX_ROUNDS baseado em dados reais de cobertura).

Coleta de métricas durante a revisão

collect_review_metrics() {
  local pr_number="$1"
  local worktree_path="$2"
  local findings_file="$3"

  # Linhas no diff
  local diff_lines
  diff_lines=$(gh pr diff "${pr_number}" | grep -c '^[+-]' 2>/dev/null || echo 0)

  # Arquivos modificados
  local files_changed
  files_changed=$(gh pr diff "${pr_number}" --name-only | wc -l)

  # Arquivos com pelo menos um finding
  local files_with_findings
  files_with_findings=$(jq -r '[.rounds[].findings[].file] | unique | length' \
    "${findings_file}")

  # Cobertura = % de arquivos modificados que têm findings
  local coverage=0
  if (( files_changed > 0 )); then
    coverage=$(( files_with_findings * 100 / files_changed ))
  fi

  # Findings por rodada
  local per_round
  per_round=$(jq -r '.rounds[] | "\(.round) (\(.persona)): \(.findings | length) findings"' \
    "${findings_file}")
