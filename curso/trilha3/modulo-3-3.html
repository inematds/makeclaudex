1

A Máquina de Estados

SETUP → REVIEWING → SUMMARIZING → DONE (e FAILED)

A máquina de estados é o esqueleto do sistema. Cada estado define o que pode acontecer e quais transições são permitidas. Estados explícitos eliminam comportamento indefinido — quando o sistema não sabe o que fazer (falha parcial, reinicialização), o estado persistido é a fonte de verdade sobre onde retomar ou o que limpar.

Diagrama de Transições

                    ┌─────────┐
   /review-pr NNN   │  SETUP  │  Cria worktree, busca contexto do PR
   ────────────────▶│         │  Inicializa estado YAML
                    └────┬────┘
                         │ worktree criado com sucesso
                         ▼
                    ┌──────────┐
     loop: round++  │REVIEWING │  Executa runner com persona da rodada
     ◀──────────────│          │◀─────────────────┐
     BLOCK (exit 2) └────┬─────┘                  │
                         │ current_round == max_rounds   │
                         ▼                         │
                    ┌───────────┐ ainda há         │
                    │SUMMARIZING│ findings? ───────┘
                    │           │ (retorna para REVIEWING)
                    └─────┬─────┘
                          │ relatório gerado + postado
                          ▼
                    ┌──────┐
                    │ DONE │  Cleanup de worktree
                    └──────┘

    Em qualquer estado → FAILED (via ERR trap)

SETUP

Cria worktree, fetch da branch, inicializa estado YAML, busca metadados do PR via gh CLI. Se falhar aqui: nenhum dano, cleanup simples.

REVIEWING

Loop de rodadas. Cada rodada incrementa current_round, seleciona persona, executa análise e acumula findings. Stop Hook decide se continua ou avança.

SUMMARIZING

Uma única rodada com persona de síntese. Consolida findings, gera relatório markdown, posta no GitHub. Transição para DONE após sucesso.

FAILED

Estado irrecuperável via ERR trap. Tenta gerar relatório parcial com findings coletados até o momento antes do cleanup.

2

O Stop Hook como Motor

BLOCK mantém o loop, APPROVE encerra — o coração do sistema

O Stop Hook é chamado pelo Claude Code ao fim de cada turno. Ele lê o estado YAML, decide o que fazer na próxima rodada e injeta instruções para Claude. O exit code 2 sinaliza BLOCK (continua), exit code 0 sinaliza APPROVE (encerra o loop).

stop-hook.sh — lógica central de decisão

#!/usr/bin/env bash

set -euo pipefail

STATE_FILE=$(find .claudex/state -name "review-PR*.yaml" \
  -newer /tmp/sentinel 2>/dev/null | head -1)

[[ -z "${STATE_FILE}" ]] && exit 0  # Nenhuma revisão ativa → APPROVE

STATUS=$(yq e '.status' "${STATE_FILE}")
CURRENT_ROUND=$(yq e '.current_round' "${STATE_FILE}")
MAX_ROUNDS=$(yq e '.max_rounds' "${STATE_FILE}")

case "${STATUS}" in
  SETUP)
    # Primeira chamada — inicia primeira rodada
    update_state status "REVIEWING"
    update_state current_round 1
    # Injeta instrução para Claude iniciar rodada 1
    echo "INSTRUÇÃO: Inicie a rodada 1 de revisão usando a persona Author."
    exit 2  # BLOCK — continua
    ;;

  REVIEWING)
    if (( CURRENT_ROUND < MAX_ROUNDS )); then
      # Ainda há rodadas — avança para próxima
      update_state current_round "$(( CURRENT_ROUND + 1 ))"
      NEXT_PERSONA=$(get_persona_for_round "$(( CURRENT_ROUND + 1 ))")
      echo "INSTRUÇÃO: Execute a rodada $(( CURRENT_ROUND + 1 )) com persona ${NEXT_PERSONA}."
      exit 2  # BLOCK — continua
    else
      # Todas as rodadas concluídas — avança para sumarização
      update_state status "SUMMARIZING"
      echo "INSTRUÇÃO: Execute a fase SUMMARIZING. Consolide todos os findings."
      exit 2  # BLOCK — continua
    fi
    ;;

  SUMMARIZING)
    # Sumarização concluída — encerra
    update_state status "DONE"
    exit 0  # APPROVE — loop encerrado
    ;;

  DONE|FAILED)
    exit 0  # APPROVE — nada mais a fazer
    ;;
esac
💡 A Mecânica do BLOCK

Quando o Stop Hook retorna exit code 2, o Claude Code não encerra a sessão. Em vez disso, qualquer texto escrito no stdout do hook é injetado como nova instrução para Claude no próximo turno. Isso é o que permite ao hook dirigir o comportamento de Claude em cada rodada — a instrução injetada define qual persona usar e qual objetivo da rodada.

3

Passando Contexto do PR ao Claude

Diff, commit messages e descrição via gh CLI

Cada rodada de análise precisa de contexto completo do PR. O runner script monta esse contexto dinamicamente antes de invocar claude --print. Contexto rico melhora drasticamente a qualidade dos findings — Claude pode verificar se a implementação atende o objetivo declarado.

Construção do contexto via gh CLI

build_pr_context() {
  local pr_number="$1"

  # Metadados do PR
  local pr_title pr_body pr_labels
  pr_title=$(gh pr view "${pr_number}" --json title -q '.title')
  pr_body=$(gh pr view "${pr_number}" --json body -q '.body')
  pr_labels=$(gh pr view "${pr_number}" --json labels -q '.labels[].name' | tr '\n' ', ')

  # Diff completo do PR
  local pr_diff
  pr_diff=$(gh pr diff "${pr_number}")

  # Histórico de commits
  local commit_log
  commit_log=$(gh pr view "${pr_number}" \
    --json commits -q '.commits[] | "[\(.oid[:7])] \(.messageHeadline)"')

  # Monta contexto para o prompt
  cat <

gh pr view

Metadados: título, descrição, labels, reviewers, milestone. Define o "porquê" do PR.

gh pr diff

O diff unificado completo de todas as mudanças. É o "o quê" do PR.

Commits

Histórico de raciocínio. Commits de revert ou refactor revelam incertezas do autor.

⚡ Dica Prática

Para PRs com diffs muito grandes (mais de 2000 linhas), considere truncar o diff e focar nos arquivos mais críticos. Use gh pr diff --name-only para listar os arquivos e git -C "${WORKTREE}" diff origin/main -- arquivo.ts para diff por arquivo.

4

Controlando Rodadas

MAX_ROUNDS configurável e estado de rodada atual

MAX_ROUNDS é a garantia de terminação do sistema. Sem ele, um loop de revisão poderia rodar indefinidamente se o Stop Hook nunca atingir a condição de APPROVE. É um parâmetro de segurança de custo, não apenas de performance.

review-config.yaml — configuração de rodadas

# .claude/review-config.yaml
max_rounds: 3                    # Default: 3 rodadas de análise
personas:
  - round: 1
    name: "Author"
    focus: "Intenção e completude"
  - round: 2
    name: "Reviewer"
    focus: "Bugs e qualidade"
  - round: 3
    name: "Security"
    focus: "Vulnerabilidades e riscos"

# Thresholds para integração com CI
blocking_severity: "HIGH"        # Findings >= HIGH bloqueiam merge
report_on_no_findings: true      # Sempre gera relatório, mesmo sem findings
💡 Garantia de Terminação

O Stop Hook verifica: se current_round >= max_rounds, força transição para SUMMARIZING independente de qualquer outra condição. Isso garante que mesmo um bug no código de transição de persona não cria um loop infinito. MAX_ROUNDS é o safety net final.

O Runner Script

Delegando análise ao Claude com contexto de worktree

O runner script é a ponte entre o Stop Hook (orquestrador) e Claude (executor). Ele constrói o prompt completo para a rodada atual — contexto do PR + instrução de persona + findings anteriores — e invoca claude --print a partir do diretório do worktree para que Claude acesse o código da branch do PR.

runner.sh — executa uma rodada de análise

run_review_round() {

⚠️ Ponto Crítico: cd no worktree

O cd "${worktree_path}" antes de invocar Claude não é opcional. Sem ele, Claude opera no diretório principal do repositório (sua branch de trabalho) em vez da branch do PR. Isso viola a garantia de isolamento e pode resultar em findings incorretos ou modificação da branch errada.

⚡ Dica Prática

Use claude --print --output-format json para obter output estruturado do Claude diretamente, sem necessidade de parsear texto livre. O output JSON facilita a extração de findings e reduz erros de parsing.

Findings Incrementais

Acumulando achados sem repetir análises das rodadas anteriores

Cada rodada de análise deve adicionar valor novo ao conjunto de findings. Para garantir isso, os findings das rodadas anteriores são passados ao prompt de cada nova rodada com instrução explícita de não duplicar. O arquivo de findings é persistido em JSON para facilitar leitura e deduplicação.

.claudex/state/findings-PR123.json — formato acumulado

{
  "pr_number": 123,
  "total_findings": 6,
  "rounds": [
    {
      "round": 1,
      "persona": "Author",
      "findings": [
        {
          "id": "f001",
          "file": "src/auth/jwt.ts",
          "line": 45,
          "severity": "HIGH",
          "type": "intent-gap",
          "description": "PR descreve expiração configurável mas expiry está hardcoded em 3600s"
        }
      ]
    },
    {
      "round": 2,
      "persona": "Reviewer",
      "findings": [
        {
          "id": "f002",
          "file": "src/auth/jwt.ts",
          "line": 78,
          "severity": "MEDIUM",
          "type": "error-handling",
          "description": "Token inválido não retorna erro específico, dificulta debugging"
        }

}
💡 A Instrução de Não-Duplicação

O prompt de cada rodada inclui a lista de findings anteriores com instrução explícita:

"Os seguintes problemas já foram identificados nas rodadas anteriores. NÃO os repita. Foque em problemas que ainda não foram cobertos pela perspectiva da sua persona: [lista de findings anteriores]"

Isso maximiza o valor por token e garante que cada rodada adiciona perspectiva genuinamente nova.
