Por que uma Única Perspectiva é Insuficiente

O viés do revisor único e o que ele perde

Um revisor experiente traz excelência em sua área de especialidade — mas também traz pontos cegos estruturais. Um engenheiro focado em qualidade de código raramente pensa em vetores de ataque. Um especialista em segurança pode não perceber que a implementação não resolve o problema descrito na task. Isso não é limitação pessoal — é a natureza da especialização.

💡 O Problema do Viés Único

Estudos em engenharia de software mostram que revisores encontram em média 60% dos defeitos quando analisam código sozinhos. Com duas perspectivas independentes, a cobertura sobe para 80%+. Três perspectivas com foco diferente cobrem sistematicamente os principais eixos de risco: intenção, qualidade e segurança.

Author encontra

  • • Implementação que não atende ao objetivo
  • • Edge cases não cobertos pela descrição
  • • Mudanças de escopo não documentadas
  • • Regressões em funcionalidades existentes

Reviewer encontra

  • • Bugs de lógica e race conditions
  • • Código duplicado e violações de DRY
  • • Cobertura de testes insuficiente
  • • Performance issues e memory leaks

Security encontra

  • • Injection vulnerabilities (SQL, XSS)
  • • Autenticação e autorização incorretas
  • • Exposição de dados sensíveis em logs
  • • Dependências com CVEs conhecidos
⚡ Dica Prática

Para repositórios de infraestrutura ou dados, adicione uma quarta persona "SRE" focada em observabilidade, alertas e falhas em produção. Para repositórios de frontend, substitua Security por "UX/Accessibility". O sistema é extensível por design via review-config.yaml.

2

As 3 Personas

Author, Reviewer e Security — papéis, foco e heurísticas

Cada persona é mais que um nome — é um conjunto de heurísticas e perguntas que direciona a análise do modelo. A definição precisa dessas heurísticas é o que diferencia uma revisão superficial de uma análise profunda.

✍️

Persona: Author

Rodada 1 — Entende intenção, valida completude

O Author conhece o código por dentro — entende por que cada decisão foi tomada. Mas isso também significa que pode ter pontos cegos em relação ao que foi prometido vs o que foi entregue.

Perguntas que o Author faz:

  • • "A implementação resolve o problema descrito no PR?"
  • • "Os edge cases mencionados na task estão cobertos?"
  • • "Há funcionalidades prometidas que não foram implementadas?"
  • • "Algo foi implementado que não estava no escopo?"
🔍

Persona: Reviewer

Rodada 2 — Encontra bugs, avalia qualidade técnica

O Reviewer experiente lê código com ceticismo saudável. Cada função é uma hipótese que precisa ser verificada, cada path de erro uma possibilidade de falha silenciosa.

Perguntas que o Reviewer faz:

  • • "O que acontece quando este valor é null/undefined?"
  • • "Esta função pode falhar silenciosamente?"
  • • "Os testes cobrem este path de código?"
  • • "Este código vai ser difícil de debugar em produção?"
🔒

Persona: Security

Rodada 3 — Encontra vulnerabilidades e riscos

O especialista em segurança assume que todos os inputs são maliciosos e que qualquer ponto de entrada é um vetor de ataque potencial.

Perguntas que o Security faz:

  • • "Como um atacante pode explorar este código?"
  • • "Dados sensíveis estão sendo logados ou expostos?"
  • • "A autorização está sendo verificada no lugar certo?"
  • • "Há possibilidade de injection neste ponto?"
3

Rotação de Personas por Rodada

Round 1 → Author, Round 2 → Reviewer, Round 3 → Security

A ordem importa. Author primeiro estabelece o contexto de intenção que as personas seguintes usam como referência. Reviewer usa o entendimento de intenção para distinguir "bug" de "design diferente do usual". Security usa o entendimento da feature para avaliar se os controles de acesso fazem sentido para o que está sendo construído.

R1

Rodada 1: Author — Contexto e Intenção

Lê descrição do PR, commits e implementação. Verifica alinhamento entre intenção e entrega. Identifica gaps de escopo. Estabelece o "contrato" que as próximas personas usam como referência.

R2

Rodada 2: Reviewer — Qualidade e Correção

Com contexto do Rodada 1, analisa qualidade da implementação. Conhecendo o objetivo, pode distinguir inconsistência de design proposital de bug real. Foca em correção técnica e testabilidade.

R3

Rodada 3: Security — Riscos e Vulnerabilidades

Com contexto de o quê a feature faz e como foi implementada, avalia riscos de segurança específicos ao domínio. Uma feature de autenticação tem riscos diferentes de uma feature de relatório.

💡 Por que Author primeiro

Se Reviewer fosse primeiro, poderia marcar como "bug" um comportamento que é intencional mas diferente do que ele esperaria. Ao saber da intenção primeiro (Author), Reviewer pode avaliar: "Dado que o objetivo é X, esta implementação é correta?" em vez de "Esta implementação é igual ao padrão que espero?".

4

Prompt Engineering para Personas

Instruções específicas por papel com exemplos few-shot

A qualidade da persona é determinada pela qualidade das suas instruções. Um prompt vago como "Você é um revisor de código" produz análise genérica. Um prompt com papel preciso, exemplos concretos e formato de output definido produz findings acionáveis.

Instrução da persona Security (exemplo)

PERSONA_SECURITY_INSTRUCTIONS = """
Você é um especialista em segurança de aplicações com 10 anos de experiência
em revisão de código em produção. Seu único foco nesta rodada é SEGURANÇA.

Você NÃO comenta sobre:
- Estilo de código ou naming conventions
- Performance ou otimizações
- Cobertura de testes (exceto se relacionado a segurança)

Você SEMPRE analisa:
- Injection vulnerabilities (SQL, NoSQL, Command, LDAP, XPath)
- Autenticação e autorização (missing checks, privilege escalation)
- Exposição de dados sensíveis (logs, respostas de API, storage)
- Cryptografia (algoritmos fracos, chaves hardcoded, IV fixo)
- Rate limiting e proteção contra abuse
- Dependências com CVEs conhecidos

Exemplos de findings que você reportaria:

FINDING CORRETO:
{
  "file": "src/api/users.ts",
  "line": 45,
  "severity": "CRITICAL",
  "type": "missing-authz",
  "description": "getUserById aceita userId do body sem verificar se o requester tem permissão para acessar este usuário. Qualquer usuário autenticado pode acessar dados de qualquer outro."
}

FINDING ERRADO (não é seu papel):
{
  "type": "style",
  "description": "Variável deveria se chamar userId em vez de id"
}

Retorne apenas JSON com array de findings.
"""

Elementos eficazes

  • ✓Papel explícito com anos de experiência
  • ✓Lista do que a persona NÃO faz
  • ✓Lista do que a persona SEMPRE faz
  • ✓Exemplos few-shot de findings corretos/errados
  • ✓Formato de output explicitamente definido

Elementos que enfraquecem

  • ✗"Revise o código como um especialista"
  • ✗Sem limitação de escopo (encontra tudo)
  • ✗Sem exemplos de findings esperados
  • ✗Formato de output aberto (texto livre)
  • ✗Instruções que se contradizem entre personas

Acumulando Findings Entre Rodadas

Como não repetir o que já foi analisado

Sem contexto das rodadas anteriores, cada persona analisa o código do zero e inevitavelmente redescobre os mesmos problemas. O sistema de acumulação de findings garante que cada rodada adiciona perspectiva genuinamente nova ao conjunto de análises.

Como os findings anteriores são passados ao próximo turno

build_round_prompt() {
  local round="$1"
  local persona="$2"
  local pr_context="$3"

Passe o sumário dos findings anteriores (arquivo + linha + descrição curta), não o JSON completo. Isso economiza tokens e mantém o foco na instrução principal. Claude entende o contexto sem precisar reprocessar o JSON detalhado de cada finding anterior.

A Persona de Síntese

Como a fase SUMMARIZING consolida todos os achados

Após todas as rodadas de análise, a fase SUMMARIZING usa uma persona especial de Senior Engineer para consolidar, deduplicar e priorizar os findings de todas as rodadas num relatório coerente e acionável. Esta persona não analisa código novo — ela sintetiza análises existentes.

Instrução da persona de síntese

PERSONA_SYNTHESIS_INSTRUCTIONS = """
Você é um Senior Engineer com visão holística do sistema. Você recebeu
findings de múltiplos revisores especializados e sua tarefa é SINTETIZAR,
não analisar código novo.

Você DEVE:
1. Consolidar findings que descrevem o mesmo problema de perspectivas diferentes
2. Priorizar por impacto real: CRITICAL > HIGH > MEDIUM > LOW > INFO
3. Adicionar contexto de como findings se relacionam entre si
4. Identificar o "tema central" dos findings (ex: "autenticação frágil em 3 pontos")
5. Sugerir ação de triagem para o autor do PR

Você NÃO DEVE:
- Analisar código que não foi coberto pelos revisores anteriores
- Mudar a severidade sem justificativa explícita
- Omitir findings mesmo que considere minor

Formato de saída: Markdown estruturado para o relatório final de revisão.
Inclua: sumário executivo, findings por severidade, recomendações de ação.
"""
💡 O Valor da Síntese

Três findings individuais sobre autenticação — um do Author (endpoint sem auth check), um do Reviewer (token não validado), um do Security (chave fraca) — são mais úteis consolidados como "O sistema de autenticação tem 3 vulnerabilidades encadeadas que juntas permitem bypass completo". A síntese revela o padrão que as análises individuais não conseguem ver.
