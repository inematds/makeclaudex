Visão Geral do Plugin

O que o revisor de PRs faz e o que o torna diferente

O Revisor de PRs com Claude Code é um plugin que automatiza a análise completa de um Pull Request: ele isola o código da branch via git worktree, executa múltiplas rodadas de análise com personas especializadas e entrega um relatório estruturado diretamente no GitHub. Diferente de um linter ou ferramenta de análise estática, ele entende intenção — a descrição do PR, os commits e o contexto da branch fazem parte da análise.

💡 Conceito Principal

Um revisor que entende por que o código existe, não apenas o que ele faz

Ferramentas de análise estática verificam sintaxe e padrões. Este plugin verifica se a implementação atende à intenção declarada no PR. Um código tecnicamente correto que não resolve o problema descrito é um achado de revisão — e só um revisor com contexto consegue identificar isso.

Isolamento

Worktree temporário em /tmp — sem toque na branch de trabalho. Cleanup garantido.

Análise Multi-perspectiva

3 rodadas com personas Author, Reviewer e Security. Findings incrementais sem repetição.

Relatório Integrado

Markdown estruturado em .claudex/reviews/ e comentário automático via gh pr comment.

⚡ Dica Prática

Inicie sempre com /review-pr 123 para um PR pequeno (menos de 200 linhas de diff) na primeira execução. PRs grandes são válidos, mas o custo e o tempo de análise aumentam proporcionalmente ao tamanho do diff.

2

Por que Revisão de PR é Mais Complexa

Múltiplos arquivos, contexto de branch, histórico de commits

Revisar um único arquivo em isolamento é simples. Revisar um PR envolve entender como múltiplos arquivos se relacionam entre si, qual é a intenção declarada nos commits, o que foi discutido anteriormente no PR e como a mudança se encaixa no contexto da codebase. Essa complexidade multidimensional é o que justifica a arquitetura em camadas do plugin.

1

Diff multi-arquivo

Um PR típico modifica 3-20 arquivos. Problemas frequentemente surgem das interações entre arquivos, não de arquivos individuais.

2

Histórico de commits

Commits individuais revelam a ordem de raciocínio do autor. Um commit que reverte uma decisão anterior é um sinal de alerta que o diff final esconde.

3

Contexto da branch de base

O PR não existe no vácuo. Arquivos não modificados podem conter código que interaje com as mudanças e precisam ser considerados.

4

Intenção declarada vs implementação real

A descrição do PR afirma o objetivo. A revisão deve verificar se a implementação realmente atinge aquele objetivo — o gap é frequentemente onde os bugs vivem.

⚠️ Atenção

Uma ferramenta que analisa apenas o diff sem contexto de branch encontra no máximo 60% dos problemas de um PR. Os outros 40% requerem entendimento de como o código novo interage com o código existente. Por isso o worktree fornece acesso completo à codebase, não apenas ao diff.

3

O Desafio do Isolamento

Por que não revisar na branch ativa

Quando Claude Code analisa código, ele pode — e frequentemente o faz — fazer modificações para entender melhor o sistema: executar scripts, criar arquivos temporários, instalar dependências. Se isso acontece na sua branch de trabalho durante uma revisão, você tem contaminação silenciosa: o código que você vai commitar pode estar diferente do que você escreveu.

Sem Isolamento

  • ✗Claude pode modificar arquivos durante análise
  • ✗Scripts de teste podem alterar estado do banco local
  • ✗Arquivos temporários ficam no seu working tree
  • ✗git status fica poluído após a revisão

Com Worktree

  • ✓Sandbox completamente isolado em /tmp
  • ✓Branch de trabalho nunca é tocada
  • ✓Worktree removido automaticamente após uso
  • ✓git status limpo garantido após revisão
💡 Conceito Principal

O isolamento é uma garantia estrutural, não uma convenção

Convenções dependem de disciplina — e sob pressão, disciplina falha. O worktree torna fisicamente impossível que a revisão afete a branch de trabalho: são diretórios diferentes no sistema de arquivos. Não importa o que Claude faça dentro do worktree, sua branch permanece intacta.

4

Arquitetura em 3 Camadas

Isolamento → Análise multi-perspectiva → Relatório estruturado

A arquitetura em 3 camadas garante que cada responsabilidade é tratada por um componente dedicado, com interfaces limpas entre eles. Uma falha na camada de relatório não impede a análise; uma falha na camada de isolamento aborta tudo antes de qualquer análise ser feita. Essa separação é o que torna o sistema seguro por default.

1

Camada de Isolamento

Componentes: worktree manager, state YAML, cleanup hook

Responsável por criar o ambiente seguro de análise. Persiste estado suficiente para cleanup garantido mesmo em falha catastrófica. Ponto de entrada e saída do sistema.

2

Camada de Análise

Componentes: Stop Hook, runner script, persona engine, findings accumulator

O loop de revisão multi-rodada. Cada rodada usa uma persona diferente e acumula findings sem repetir análises anteriores. Controlada pela máquina de estados.

3

Camada de Relatório

Componentes: synthesis persona, markdown formatter, gh pr comment

Transforma findings brutos de múltiplas rodadas num relatório coeso, priorizando por severidade e postando no GitHub. Executada na fase SUMMARIZING.

📊 Separação de Responsabilidades

Cada camada só conhece sua própria responsabilidade. A camada de análise não sabe como o relatório será formatado. A camada de relatório não sabe quantas rodadas houve. Isso é possível porque o estado YAML é o único canal de comunicação entre camadas.

5

Comparando com o Claudex

O que é idêntico e o que é diferente

Se você concluiu a Trilha 2, boa parte desta trilha será reconhecimento de padrões. A máquina de estados, o Stop Hook como motor de loop e o conceito de personas rotativas são idênticos ao Claudex. O que muda é o contexto de operação: em vez de um arquivo de código em revisão, temos uma branch inteira de PR.

Idêntico ao Claudex

  • ≡Máquina de estados com fases explícitas
  • ≡Stop Hook com BLOCK/APPROVE
  • ≡Rotação de personas por rodada
  • ≡Estado YAML com escrita atômica
  • ≡Fase SUMMARIZING que consolida findings
  • ≡Primitivas de safety: fail-open, cleanup

Novo neste projeto

  • +Git worktree para isolamento de branch
  • +gh CLI para contexto de PR (diff, commits)
  • +Personas Author/Reviewer/Security (vs Eng/Sec/SRE)
  • +Relatório persistido por número de PR
  • +Comentário automático no GitHub
  • +review-config.yaml por repositório
⚡ Dica Prática

Se você construiu o Claudex da Trilha 2, pode reusar o hook de Stop, a lógica de máquina de estados e o sistema de personas com mínimas adaptações. O delta real de trabalho novo está na camada de isolamento (worktree) e integração com GitHub (gh CLI).

6

O SPEC do Projeto

As 8 garantias de segurança e o blueprint de arquivos

Antes de qualquer código, o SPEC deste projeto define 8 garantias de segurança que devem ser verdadeiras em qualquer estado do sistema. Quatro são herdadas do Claudex (fail-open, estado atômico, CAS, cleanup garantido); quatro são novas e específicas para o contexto de revisão de PRs em repositórios de produção.

Herdada 1

Fail-open

Qualquer falha no plugin libera Claude Code para continuar trabalhando normalmente.

Herdada 2

Estado Atômico

O arquivo de estado nunca fica em estado corrompido ou parcialmente escrito.

Herdada 3

CAS (Compare-and-Swap)

Transições de estado verificam a versão atual antes de escrever a nova.

Herdada 4

Cleanup Garantido

Recursos temporários são removidos em qualquer saída, incluindo SIGTERM.

Nova 5

Branch Intocada

A branch de trabalho do desenvolvedor nunca é modificada durante a revisão.

Nova 6

Relatório Sempre Gerado

Mesmo em falha parcial, um relatório mínimo com o que foi encontrado é gerado e salvo.

Nova 7

Worktree Orphan Prevention

Sistema detecta e remove worktrees órfãos de revisões anteriores antes de criar novo.

Nova 8

Terminação Garantida

MAX_ROUNDS garante que nenhuma revisão corre indefinidamente, independente do estado.

Blueprint de Arquivos

│   ├── runner.sh            # Invoca claude --print por rodada
│   └── report.sh            # Gera e posta relatório
└── review-config.yaml       # Configuração por repositório

└── reviews/
    ├── PR-123.md            # Relatório persistido por PR
    └── PR-124.md
