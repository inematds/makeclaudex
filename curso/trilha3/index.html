🎯 O Projeto: Revisor de PRs com Claude Code

Visão geral do plugin, arquitetura em 3 camadas e comparação com o Claudex da Trilha 2.

O que é:

Um plugin de revisão de PRs que automatiza a análise de código de branches inteiras, usando isolamento de worktree, múltiplas rodadas e personas especializadas.

Por que aprender:

Revisões manuais são inconsistentes e dependem do humor do revisor. Este plugin garante cobertura sistemática com múltiplas perspectivas em cada PR.

Conceitos-chave:

Plugin de revisão, automação de PR, análise multi-rodada, integração com gh CLI.

O que é:

A complexidade adicional de revisar PRs versus código inline: múltiplos arquivos modificados, contexto de commits, descrição de PR e histórico de discussões.

Por que aprender:

Entender essa complexidade justifica as escolhas arquiteturais — worktree, gh CLI e múltiplas rodadas — que parecem excessivas sem este contexto.

Conceitos-chave:

Diff de PR, commit messages, contexto de branch, escopo de revisão, surface area de análise.

O que é:

O risco de modificação acidental quando Claude Code analisa código na branch de trabalho ativa — e como git worktree resolve esse problema estruturalmente.

Por que aprender:

Sem isolamento, um bug no plugin pode modificar o código que está sendo revisado, contaminando o resultado ou danificando a branch de produção.

Conceitos-chave:

Isolamento de ambiente, risco de contaminação, git worktree como sandbox, separação entre revisor e código revisado.

O que é:

As três camadas separadas que compõem o sistema: Isolamento (worktree + YAML de estado), Análise multi-perspectiva (loop de personas) e Relatório estruturado (markdown + gh CLI).

Por que aprender:

Separar responsabilidades em camadas permite testar, depurar e evoluir cada parte independentemente. Um bug na camada de relatório não afeta a análise.

Conceitos-chave:

Separação de camadas, responsabilidade única, isolamento por camada, testabilidade modular.

O que é:

Mapeamento explícito entre os dois sistemas: máquina de estados e personas são idênticos; worktree, gh CLI e relatório de PR são novos.

Por que aprender:

Identificar o que é reaproveitável reduz o esforço de build. Você não está começando do zero — está estendendo padrões que já conhece.

Conceitos-chave:

Reúso de padrões, delta de complexidade, extensão vs reescrita, transferência de conhecimento entre projetos.

O que é:

O documento SPEC deste projeto com 8 garantias de segurança específicas para revisão de PR: worktree cleanup garantido, sem modificação na branch de trabalho, relatório sempre gerado mesmo em falha.

Por que aprender:

Garantias de segurança mais rígidas são necessárias porque este sistema opera em repositórios de produção. Um erro aqui pode afetar código de outros desenvolvedores.

Conceitos-chave:

SPEC de segurança, garantias de worktree, blueprint de arquivos, invariantes de produção.

🌳 Isolamento de Branch com Git Worktree

Como criar checkouts isolados, gerenciar estado do worktree em YAML e garantir limpeza em qualquer saída.

O que é:

Um recurso do Git que permite ter múltiplos diretórios de trabalho do mesmo repositório ativos simultaneamente, cada um com sua própria branch e estado de arquivos.

Por que aprender:

É a primitiva fundamental que permite ao revisor operar na branch do PR sem jamais tocar na branch onde o desenvolvedor está trabalhando.

Conceitos-chave:

git worktree, checkout simultâneo, linked worktree, HEAD separado por diretório.

O que é:

O comando git worktree add /tmp/review-XYZ origin/feature-branch que cria um diretório temporário com a branch do PR já carregada.

Por que aprender:

O path dinâmico (com timestamp ou PR number) garante que cada revisão usa um diretório único, evitando conflitos entre revisões paralelas.

Conceitos-chave:

git worktree add, path temporário, fetch antes de criar, tracking de branch remota.

O que é:

O hook de cleanup que executa git worktree remove --force garantidamente em qualquer saída — sucesso, falha ou sinal de interrupção (SIGTERM, SIGINT).

Por que aprender:

Worktrees órfãos acumulam espaço em disco e podem causar conflitos em revisões futuras. Cleanup garantido é uma invariante de segurança, não uma feature opcional.

Conceitos-chave:

trap ERR EXIT, git worktree remove, cleanup atômico, handler de sinal.

O que é:

O arquivo de estado que persiste o caminho do worktree, nome da branch, número do PR e status atual, permitindo retomada e cleanup mesmo após reinicialização.

Por que aprender:

Sem estado persistido, uma falha no meio da revisão deixa o worktree órfão. Com YAML de estado, o sistema sabe o que limpar mesmo se reiniciado.

Conceitos-chave:

State YAML, worktree path tracking, PR metadata, status machine, escrita atômica.

O que é:

A verificação e resolução quando o worktree já existe no path de destino — seja por uma revisão anterior que não foi limpa ou por uma revisão paralela em andamento.

Por que aprender:

Tentar criar um worktree onde já existe um falha silenciosamente ou com erro críptico. O fail-safe torna o comportamento determinístico e auditável.

Conceitos-chave:

Worktree lock file, detecção de existência, estratégia de resolução, idempotência de criação.

O que é:

O conjunto de testes que verifica o isolamento real: modificar um arquivo no worktree e confirmar que a branch principal permanece intacta via git diff.

Por que aprender:

Isolamento é a garantia fundamental do sistema. Testá-lo explicitamente demonstra que a invariante é real e não apenas assumida.

Conceitos-chave:

Teste de isolamento, git status por diretório, verificação de estado limpo, smoke test de worktree.

🔄 O Loop de Revisão

A máquina de estados SETUP→REVIEWING→SUMMARIZING→DONE, o Stop Hook como motor e findings incrementais.

O que é:

A máquina de estados de 4 fases que governa o ciclo de vida completo de uma revisão de PR, incluindo o estado FAILED para erros irrecuperáveis.

Por que aprender:

Estados explícitos eliminam ambiguidade sobre o que o sistema está fazendo e o que fazer em cada situação, especialmente em falhas parciais.

Conceitos-chave:

State machine, fases de revisão, transições permitidas, estado FAILED, persistência de estado.

O que é:

O Stop Hook que intercepta cada fim de turno e decide: exit code 2 (BLOCK, continua loop) ou exit code 0 (APPROVE, encerra) com base no estado atual da revisão.

Por que aprender:

O hook é o único lugar onde o controle de fluxo é determinístico. Entender sua lógica de decisão é entender o coração do sistema.

Conceitos-chave:

Stop Hook, exit codes, leitura de estado, injeção de instrução, controle de loop.

O que é:

A estratégia de injeção de contexto: diff do PR via gh pr diff, mensagens de commit, descrição e labels do PR são incluídos no prompt de revisão de cada rodada.

Por que aprender:

Claude precisa de contexto completo para revisar efetivamente. Sem a intenção do PR (descrição, commits), a revisão vira análise estática sem entender o objetivo.

Conceitos-chave:

gh pr diff, gh pr view, context injection, prompt engineering com contexto dinâmico.

O que é:

O mecanismo de controle de rodadas: variável MAX_ROUNDS por configuração, contador de rodada atual no estado YAML, transição automática para SUMMARIZING quando o limite é atingido.

Por que aprender:

Sem limite explícito, o loop pode rodar indefinidamente consumindo tokens. MAX_ROUNDS é uma garantia de custo, não apenas uma preferência de usuário.

Conceitos-chave:

MAX_ROUNDS, current_round counter, terminação garantida, transição automática de fase.

O que é:

O script que invoca claude --print com o prompt de revisão correto para cada rodada, operando dentro do diretório do worktree para acessar o código da branch do PR.

Por que aprender:

O runner é a ponte entre o orquestrador (hook) e o executor (Claude). Seu design determina a qualidade do contexto que Claude recebe em cada rodada.

Conceitos-chave:

claude --print, subprocess com cwd, context building, output capture e parsing.

O que é:

O mecanismo de acumulação de findings entre rodadas: cada nova rodada recebe os findings das rodadas anteriores como contexto para não duplicar análises já feitas.

Por que aprender:

Sem acumulação inteligente, cada rodada repetiria os mesmos achados. O sistema incremental garante que cada rodada adiciona valor novo ao resultado.

Conceitos-chave:

Findings acumulados, deduplicação, contexto de rodadas anteriores, eficiência de token.

👁️ Personas de Revisão Multi-perspectiva

Author, Reviewer e Security como perspectivas rotativas — prompt engineering e acumulação de findings entre rodadas.

O que é:

O problema fundamental de revisar código com uma única perspectiva: cada papel (autor, revisor técnico, especialista em segurança) encontra problemas diferentes no mesmo código.

Por que aprender:

Justifica a complexidade adicional do sistema de personas. Sem esse contexto, as 3 rodadas parecem redundantes em vez de complementares.

Conceitos-chave:

Viés cognitivo de revisor, perspectiva especializada, cobertura de análise, blind spots por papel.

O que é:

As três personas distintas: Author (entende intenção, valida implementação vs objetivo), Reviewer (encontra bugs e problemas de qualidade), Security (encontra vulnerabilidades e riscos).

Por que aprender:

Cada persona ativa um conjunto diferente de heurísticas no modelo. A definição precisa das personas determina a qualidade dos findings em cada rodada.

Conceitos-chave:

Author persona, Reviewer persona, Security persona, heurísticas por papel, instruções específicas.

O que é:

O mecanismo de seleção de persona por número de rodada: rodada 1 usa Author, rodada 2 usa Reviewer, rodada 3 usa Security — configurável via review-config.yaml.

Por que aprender:

A ordem importa: Author primeiro estabelece o contexto de intenção que as personas seguintes usam para avaliar se a implementação está correta para o objetivo declarado.

Conceitos-chave:

Persona rotation schedule, round-to-persona mapping, ordem de análise, configuração de rotação.

O que é:

A técnica de construir system prompts específicos para cada persona, com exemplos concretos de findings esperados e formato de output exigido.

Por que aprender:

Um prompt de persona vago produz análise genérica. Instruções precisas com exemplos ancoram o modelo no papel correto e no formato esperado.

Conceitos-chave:

System prompt de persona, exemplos few-shot, formato de output, ancoragem de papel.

O que é:

O sistema de passagem de findings anteriores para cada nova rodada, instruindo explicitamente a persona atual a focar em aspectos ainda não cobertos pelas rodadas anteriores.

Por que aprender:

Sem contexto de rodadas anteriores, cada persona analisa o código do zero e pode duplicar findings. O acúmulo incremental maximiza a cobertura por token gasto.

Conceitos-chave:

Findings acumulados, instrução de não-duplicação, contexto cross-round, eficiência de análise.

O que é:

A fase SUMMARIZING que usa uma persona especial de "Senior Engineer" para consolidar, deduplicar e priorizar findings de todas as rodadas num relatório coerente e acionável.

Por que aprender:

Findings individuais de múltiplas rodadas podem ser contraditórios ou redundantes. A persona de síntese transforma dados brutos em recomendações priorizadas.

Conceitos-chave:

Synthesis persona, consolidação de findings, priorização, deduplicação, relatório final coerente.

📝 Relatório de Revisão

Formato markdown estruturado, severidade de findings, agrupamento por arquivo e integração com GitHub via gh pr comment.

O que é:

A estrutura markdown padronizada do relatório: cabeçalho com metadados do PR, seções por severidade, findings individuais com arquivo/linha/descrição e seção de sumário executivo.

Por que aprender:

Um formato consistente permite automação — CI/CD pode parsear o relatório, contar findings críticos e bloquear merge automaticamente.

Conceitos-chave:

Markdown estruturado, parseable output, headers consistentes, frontmatter de metadados.

O que é:

O sistema de 5 níveis de severidade com critérios explícitos: CRITICAL (bloqueador de merge), HIGH (deve ser corrigido antes de deploy), MEDIUM/LOW (recomendações), INFO (observações).

Por que aprender:

Severidade sem critérios claros é arbitrária. Com critérios definidos, a mesma issue será classificada igual por qualquer persona em qualquer rodada.

Conceitos-chave:

Severity taxonomy, critérios de classificação, blocking vs non-blocking, critério de merge gate.

O que é:

A lógica de organização do relatório final que agrupa findings por arquivo modificado e por tipo (segurança, qualidade, performance), facilitando a triagem por desenvolvedor.

Por que aprender:

Findings em ordem cronológica de análise são difíceis de triagem. Agrupamento por arquivo permite ao desenvolvedor abrir um arquivo e resolver todos os problemas de uma vez.

Conceitos-chave:

Finding grouping, por arquivo, por tipo, merge sort de findings, estrutura de triagem.

O que é:

A convenção de salvar cada relatório em .claudex/reviews/PR-NNN.md com nomenclatura baseada no número do PR, criando um histórico auditável de revisões.

Por que aprender:

Relatórios persistidos permitem comparar revisões do mesmo PR ao longo do tempo (após novas iterações) e servem como documentação de decisões de engenharia.

Conceitos-chave:

Artefato persistente, naming convention, histórico de revisões, documentação de decisão.

O que é:

O passo final que posta o relatório como comentário no PR via gh pr comment --body-file, tornando os findings visíveis diretamente na interface do GitHub.

Por que aprender:

Findings que ficam só em arquivo local têm impacto zero. A integração com GitHub coloca os findings onde o desenvolvedor já está — na thread do PR.

Conceitos-chave:

gh pr comment, body-file, API do GitHub, visibilidade de findings, notificação automática.

O que é:

As métricas incluídas no relatório: total de linhas no diff, número de findings por rodada e por severidade, percentual de arquivos com ao menos um finding.

Por que aprender:

Métricas permitem comparar qualidade de revisão entre PRs, identificar se o número de rodadas está sendo suficiente e calibrar MAX_ROUNDS para o repositório.

Conceitos-chave:

Revisão métricas, diff coverage, finding density, calibração de configuração.

🏭 Produção e Próximos Passos

Safety pass, rate limiting, configuração por repositório e o que separa hobby de produção.

O que é:

A aplicação das 6 primitivas de segurança da Trilha 2 ao novo contexto: fail-open, cleanup garantido, estado atômico, idempotência de criação de worktree, timeout de revisão e limpeza de artefatos temporários.

Por que aprender:

As primitivas são as mesmas, mas seus pontos de aplicação mudaram. Identificar onde aplicar cada uma no novo sistema é o exercício de maturidade de engenharia desta trilha.

Conceitos-chave:

Safety primitives, aplicação contextual, mapeamento de garantias, auditoria de segurança.

O que é:

A estratégia de teste end-to-end usando um PR de teste sintético com issues intencionais: um bug, uma vulnerabilidade de segurança e um problema de performance — para validar que cada persona encontra o que deveria.

Por que aprender:

Testar com PRs reais de produção é arriscado. Um PR de teste sintético com issues conhecidas permite validar a cobertura das personas de forma determinística.

Conceitos-chave:

Teste E2E, PR sintético, golden path testing, validação de cobertura de personas.

O que é:

O modelo de custo do plugin: estimativa de tokens por rodada baseada no tamanho do diff, custo por PR com 3 rodadas, e configurações para controlar o gasto em repositórios de alto volume.

Por que aprender:

Um plugin sem controle de custo pode gerar faturas inesperadas em repos com muitos PRs. Estimativa e limites de custo são features de produção, não luxos.

Conceitos-chave:

Token budget, custo por rodada, MAX_TOKENS por chamada, throttling de revisões, alertas de custo.

O que é:

O arquivo de configuração por repositório que define MAX_ROUNDS, personas ativas, threshold de severidade para bloquear merge e padrões de arquivos a ignorar na análise.

Por que aprender:

Cada repositório tem perfil de risco diferente. Um repo de serviço financeiro precisa de Security como primeira persona; um repo de frontend pode priorizar Reviewer.

Conceitos-chave:

review-config.yaml, configuração por repo, override de padrões, persona order, severity threshold.

O que é:

As três dimensões que elevam um script funcional a produção: observabilidade (logs estruturados, métricas de revisão), rollback (desfazer comentário postado) e alertas (falha silenciosa detectada).

Por que aprender:

Um sistema que funciona 95% do tempo e falha silenciosamente nos outros 5% é mais perigoso que um sistema que não existe. Produção exige detectabilidade de falha.

Conceitos-chave:

Observabilidade, logs estruturados, rollback de comentário, alertas de falha silenciosa.

O que é:

As três extensões naturais do sistema: integração com CI/CD (rodar na abertura de PR), revisão multi-model (usar modelos diferentes por persona) e auto-apply de fixes para issues LOW/INFO.

Por que aprender:

O sistema atual é manual (iniciado pelo desenvolvedor). As extensões tornam-no proativo e totalmente integrado ao workflow de engenharia da empresa.

Conceitos-chave:

CI integration, GitHub Actions, multi-model review, auto-fix, workflow proativo.
